> **EstreGenesis 2.0 module.** Constellation graduates multi-agent coordination from the file-based `.agent/_coordination/` ledger (STATE/HANDOFF/CHANGELOG) to a **real-time live board** ظ¤ a WebSocket server + A2A (agent-to-agent) messaging + a dashboard a human watches. It is a runtime system (server + bridges + watchers); this guide distills the **whole contract inline** ظ¤ the A2A bridge interface (┬د2), the protocol essentials (┬د2ظô┬د5), and the runtime patterns (┬د4) ظ¤ and ships the runtime as **distilled `.eux` specs** in `constellation/`, so any project can build a compatible board **without access to a private reference runtime**.
>
> **Optional.** File-based coordination (the seed's Phase 5) is the default and is enough for most projects. Adopt Constellation only when concurrent multi-agent operation needs real-time visibility, live cross-agent messaging, or orchestrated delegation.
>
> **Self-sufficient.** Everything needed to implement a compatible board is in this file + the `constellation/*.eux` distillations ظ¤ build from the spec, not from a private runtime. The canonical live-board protocol is maintained upstream; this is its public distillation (v0.3) and the build-from-spec source of truth for adopters. Goal: Constellation matures toward a published EstreGenesis Claude plugin; until then it ships as a 2.0-included module.

## 1. Architecture

```
upstream agents          local IDE main agent              other local IDE agents
(API / gateway)     ظ¤ظû║   (orchestrator / supervisor)  ظ¤ظû║   (workers: other CLI/IDE coding agents, new chats)
                                    ظû▓
                          external collab agents (join URL)
```

- **main** ظ¤ orchestrator; the priority recipient of target-unspecified messages. Default = the local IDE bridge in the environment that started the live-board server (`main-agent` by default; override via `WS_PRIMARY_AGENT` env). Reassignable by graceful handoff (┬د2).
- **local** ظ¤ other local IDE agents (workers). A new chat in the same IDE is a new agent.
- **upstream** ظ¤ upstream agents; connect with a main-issued registration key (`uk-`) ظْ `upstream` role.
- **collab** ظ¤ external collaborators; connect with a collab key (`ck-`) via a join URL ظْ `collab` role, `group:collab`.
- **board** ظ¤ the dashboard / live board itself. A board sends **no** `HELLO` (the server registers HELLO senders as agents) ظ¤ it connects, receives, renders, and forwards user input as `CUSTOM`.

> **Main vs worker self-judgment (1 second, first task):** "Did I start the live-board server in this environment?" Yes = main (own the board `state.json`, onboard workers, delegate). No (server already up ظ¤ `curl :PORT/api/state` returns 200) = worker (separate `agentId` + separate queue; never write the board directly; report to main).

## 2. The A2A bridge interface (the invariant)

This is the part every adopter must describe identically. Everything else (implementation language, transport details, dashboard styling) flexes per agent/user preference.

**Roles**: `board` ┬╖ `main` ┬╖ `local` ┬╖ `upstream` (`uk-` key) ┬╖ `collab` (`ck-` key + join URL). The server decides the role from `HELLO` + the connection key; the default for a plain `HELLO` (no key, not main) is `local`.

**Connection paths** (the auth/role param is in the WS URL, not a generic `?key=`):
- dev / local: `ws://<host>:<port>/ws` (no auth by default)
- token-gated: `ws://ظخ/ws?token=<token>` (or `Authorization: Bearer`)

> **Secure-by-default bind (v2.4.11).** Reference ه£ن▓نè¤ board(نîهï£ن│┤نô£) هù░م▓░هإ *نش┤هإ╕هخإ trusted-operator* نة£ ه╖ذم╕ëوـ┤ وéج ن░£م╕ë┬╖`SetMain` هإ وùêهأروـ£نïج. ن¤░نإ╝ه£ reference `server.cjs` نè¤ **م╕░ن│╕هبه£╝نة£ loopback(`127.0.0.1`)هùندî ن░¤هإ╕نô£**وـ£نïج ظ¤ نجوè╕هؤîوش نà╕ه╢£هإ `WS_BIND=0.0.0.0`(نءنè¤ هإ╕و░وءهإ┤هèج IP) نزàهï£ opt-in هإ┤نر░, ن╣-loopback ن░¤هإ╕نô£ هï£ startup م▓╜م│بنح╝ نé╕نïج. One WS text frame = one JSON event (raw WebSocket, RFC 6455 ظ¤ no library required).

**Handshake**:
1. WS connect (per the path above) ظْ server sends `SERVER_HELLO { sessionId, protocolVersion: "0.3", serverTime }` ظْ `AgentList` (role first, so monitors can classify) ظْ `History` (replay).
2. Send `HELLO { agentId, agentName, role, capabilities{inbound[],outbound[]} }`. New IDE chat = new `agentId`. (A board sends no HELLO.)
3. Send a **`CUSTOM` event with `name="AgentHello"`** to introduce yourself. *Opt-in broadcast presence ظ¤ see ┬د13.9.1 for the normative status: AgentHello is NOT part of the required handshake (the wire-level invariant is `SERVER_HELLO` followed by `HELLO` only); it adds display metadata + triggers receiver-bridge `OnboardAck` auto-emit, which adopters MAY skip in headless / stealth / telemetry-only deployments. The ┬د2 walkthrough sends it because it's the recommended interactive default, not because the protocol requires it.* Exact wire shape ظ¤ write the literal frame; **do NOT interpret the `CUSTOM/Name` slash shorthand** used elsewhere as a human-facing label as the on-wire structure:

   ```jsonc
   { "type": "CUSTOM",
     "name": "AgentHello",
     "targetAgentId": "<main agentId>",
     "value": { "agentId": "<self agentId>",
                "agentName": "<display name>",
                "role": "local" | "upstream" | "collab",
                "env": "<runtime hint>",
                "capabilities": [ "..." ],
                "idle": true } }
   ```

   Both `name` and `targetAgentId` live at the **top level** (siblings of `type`), **not nested in `value`**. A misread of the slash shorthand has been observed to nest both inside `value`, which silently breaks (a) server routing (it reads top-level `targetAgentId`), (b) bridge auto-`OnboardAck` (it reads top-level `name === "AgentHello"`), and (c) dashboard A2A classification (it reads top-level `targetAgentId`/`agentId`) ظ¤ all three simultaneously, with no error. The slash shorthand is fine in prose; just never let it leak into emit-side code.

   The `role` field above is a **self-report hint**, not a routing source. The server takes it as a `roleHint` but the *authoritative* role classification combines the connection key (`uk-` / `ck-` / none) + the current main's resolution, and is broadcast on every `AgentList` update. Downstream routing should consume `AgentList` to learn an agent's role, not the AgentHello `value.role` field directly ظ¤ treat it as a label for the human reader and for the server's initial classification, not as truth.
4. Main replies `OnboardAck` (welcome/guide/modes/policy). Then **wait for `Delegate`** ظ¤ workers do not self-start (`Delegate` is never automated; the PM decides from the worklist).

**Messaging**:
- **Target-unspecified** inbound/CUSTOM ظْ **main** (sole priority receiver; not broadcast to all agents ظ¤ avoids duplicate processing). Main ظْ worker uses `targetAgentId` (with `reason`/`summary` meta).
- **Channel key = `agentId`** (one board tab per agent). `channelId`/`threadId` do not split channels ظ¤ they show as an origin badge / filter only.
- Workers report progress via `CUSTOM/WorkerReport`; the board (`state.json`) SSoT is the **main** ظ¤ workers never write the board directly (avoids multi-session collision).
- Outbound events follow AG-UI `UPPER_SNAKE_CASE`, broadcast to all boards tagged by `agentId`: `RUN_STARTED/FINISHED` ┬╖ `STEP_STARTED/FINISHED` ┬╖ `TEXT_MESSAGE_START/CONTENT/END` ┬╖ `TOOL_CALL_START/ARGS/END/RESULT` ┬╖ `CUSTOM`.
- Inbound (board/main ظْ agent): `CUSTOM` + name ظ¤ `UserPrompt` ┬╖ `Command{pause/resume}` ┬╖ `Cancel` ┬╖ `Delegate` ┬╖ `OnboardAck`. Queue inbound; drain at a safe point (turn boundary) = near-real-time injection.
- **A2A replies** carry a reply envelope so the board pairs requestظ¤response: `targetAgentId` = the original sender, echoed `contextId` (or `threadId`), `parentId` = the request's `messageId`. When an adapter can't echo the envelope, the server falls back to a **time-based reply-window** (most-recent request peer) ظ¤ a last resort, fragile under concurrency/latency, so echo the envelope whenever you can.
- **Telemetry exclusion**: observation-only telemetry (a low-frequency watcher's heartbeats/snapshots) is tagged so the server keeps it on board broadcast/history but excludes it from both A2A reply-window pairing **and** main routing ظ¤ the main must never mistake telemetry for user input.

**Graceful main handoff**: board `CUSTOM/SetMain{agentId}` ظْ server asks current main `HandoffRequested{to}` ظْ current main finishes in-flight, replies `HandoffReady{summary}` (or **10s timeout** force) ظْ `MainChanged` + `AgentList` push. Server uninterrupted; only role/authority transfers. Abnormal main disconnect ظْ election fallback (bridge-priority).

## 3. Operating modes

The live board's behavior is governed by modes; the SSoT is the board's `state.json` (`.modes` + the top-level `standby` field), not docs.

| mode | ON | OFF |
|---|---|---|
| `liveBoard` | server running, agents connect | not running ظ¤ agents do direct work only |
| `wsRealtime` | WS realtime (chat ┬╖ A2A ┬╖ monitor) | board SSE/MCP read only |
| `autopilot` | main self-drives (consumes the worklist) | main acts only on user instruction |
| `standby` (infinite-wait) | hold connection/poll even when idle | finish in-flight, then wait for prompt |
| `newAgentAutoJoin` | new local agent auto-joins on first task | new agents stay unattached unless told |

`standby` toggles by user request or the dashboard standby switch. The main itself defaults to autopilot (it hosts the server).

## 4. Runtime patterns (how a turn-based agent stays connected)

> **Watch-state invariant.** While joined to a board, an agent **never ends its conversation** ظ¤ it always returns to watch. A turn that finishes its work is not done; it must re-establish presence (re-arm the watcher or re-enter the wait window) before yielding. Ending the conversation while joined = going dark on the board.

Constellation needs near-real-time (ظëج15s) presence, not tick-level realtime. Two monitor patterns:

- **(A) turn-held monitor** ظ¤ runtimes that hold a long turn (API sessions, long-running CLI agents): inside the turn, repeat a ظëج15s wait window draining the inbox cursor; process on event, then wait again; respond to `ping` directly in-turn.
- **(B) self-wake watcher** ظ¤ runtimes whose turn ends (Claude Code IDE): after the turn, an external watcher polls inbox/feedback; on a meaningful change it **exits** ظْ that exit wakes the next turn (self-wake). The woken agent processes, advances its cursor, then **re-arms the watcher**.

**Role separation** (run as separate detached processes):
- **bridge daemon** ظ¤ holds the WS connection; `HELLO`/A2A; inbound queue (inbox file); **explicit** outbox emit. Control/A2A only ظ¤ it does NOT auto-capture the runtime's tool calls (`TOOL_CALL_*`/`RUN_*` appear only if the agent mirrors them to the outbox, or uses a direct-WS adapter).
- **monitor** ظ¤ drains the inbox cursor (pattern A or B).
- **watchdog** ظ¤ auto-restarts server/bridge on death; separate from the monitor.

**Three monitoring options** for instrumenting an autonomous tool loop: (1) **explicit emit** ظ¤ keep the bridge, the agent appends progress/tool calls to the outbox at safe points; (2) **runtime WS adapter** ظ¤ embed a WS client (the `gateway-client.eux` shape) inside the tool-calling loop to instrument it directly; (3) **near-realtime watcher** ظ¤ a ~15s polling telemetry process (tagged for telemetry exclusion, ┬د2). A monitor exposes a few independent switches: poll interval (default 15s), display heartbeat (default 60s, 0=off), auto-pong (default off), and routing exclusion (telemetry, ┬د2).

**Watch-state discipline** (confirm every turn before yielding, for self-wake runtimes especially):
- [ ] **Re-arm the watcher for real** ظ¤ actually relaunch the background watcher after processing. Writing "re-armed" without the real call silently stops monitoring (a real incident missed a delegation this way). This is the hard invariant.
- [ ] **Reply at least one line** over WS to any user/main prompt ظ¤ the human is watching the board; silence reads as a stall.
- [ ] **Register review items** in the board state (`decisions`) with summary + rationale + report path, so the main can review/commit.
- [ ] **Silent-disable WARN** ظ¤ any optional input that gates a watcher / handler / replay path MUST emit a WARN line at the point it's resolved to null (e.g. `[WARN] feedback=null ظ¤ events will NOT wake this watcher`). An *alive-but-deaf* watcher is as bad as no watcher; one log line saves hours of "looks healthy but isn't" diagnosis. Symmetric to the watch-state invariant ظ¤ never silent on a wake-trigger being dead.
- [ ] **Autonomous execution (absolute)** ظ¤ when the next step is already defined (a `Phase` ordering, the `planned` track, a `blocked` clearance, an in-order retire queue), **proceed in order without asking** ظ¤ pausing to confirm a defined-next-step is itself a violation of autonomous operation (which is *the* reason for adopting Constellation / Superscalar / EstreGenesis in the first place). Confirm only on: (a) loss / external publish (push ┬╖ deploy ┬╖ send ┬╖ delete), (b) a new major branch (RRP / design decision ظ¤ at the *decision point* only; the resulting `Phase A/B/C` plan is *decided execution*, not a re-gate), (c) restart-requiring deploys (apply autonomously, coordinate the *restart timing* only), (d) explicit user steering. A real misread: "RRP done ظْ PM Phase plan set ظْ 'should I start Phase A?'" ظ¤ that's mistaking the just-closed RRP gate for a new gate. Phase A is decided execution; start it.

**Residency checklist**:
- [ ] Separate `agentId`/`threadId`/`inbox`/`outbox` per worker (never occupy the main's default queue ظ¤ an unseparated thread shows the worker as the main).
- [ ] Detached residency (survive shell/session end): POSIX `setsid`/`nohup`; Windows `Start-Process -WindowStyle Hidden` + stdout/stderr files.
- [ ] Send only after `connected` (the bridge sets the outbox cursor to EOF on start; pre-connect appends are lost) ظ¤ send `AgentHello`/reports only once the connection is confirmed.
- [ ] Track a processed-line cursor; advance it after waking + processing.
- [ ] blocklist semantic filter (absorb known noise, wake on everything else) ظ¤ an allowlist drops new main-reply names (`Delegate`/`OnboardAck`/`WorkerAck`) and misses wakeups.
- [ ] heartbeat off / auto-pong off by default.
- [ ] Server-code changes don't reach a resident server until restart ظ¤ pre-announce a `ServerNotice` and let the main coordinate timing.

## 5. External collab onboarding (join URL)

External collaborators join with one URL:

1. Main issues a key: `RegisterCollabKey{label}` ظْ `CollabKeyIssued{joinUrl}`.
2. The joining agent opens `http://<host>:<port>/join/collab?key=<ck-ظخ>` ظْ receives an onboarding markdown (key/host embedded; invalid key = 403).
3. Connect with the one-line `ws://<host>:<port>/ws?key=<ck-ظخ>` ظْ `collab` role ┬╖ `group:collab`.
4. Operate per type: IDE/CLI = ┬د4 infinite-wait (bridge + self-wake watcher / turn-held monitor) ظْ wait for main delegation. Autonomous runtime = a gateway WS adapter (`constellation/gateway-client.eux`).
5. Keys are revocable (`RevokeCollabKey`).

## 8. MCP integration ظ¤ board as a Model Context Protocol surface (v0.4)

Constellation has always carried an MCP affordance ظ¤ `┬د3` operating modes maps `wsRealtime OFF` to "board SSE/MCP read only", on the design premise that an agent without a live WS connection should still be able to **read** board state through MCP. v0.4 makes that affordance explicit and extends it to **write** (A2A relay) so an external Claude Code session can join the live board through MCP as a first-class channel alongside the existing direct-WS gateway-client path.

### 8.1 Why MCP

The direct-WS path (`┬د2` A2A bridge + `gateway-client.eux` v2.5.3 SSoT-promoted) is the **runtime contract** ظ¤ agents that run continuously and need full A2A reliability use the WS connection. The MCP path is the **integration contract** ظ¤ Claude Code sessions and other MCP-capable hosts that want to *read* board state or *emit* targeted A2A messages without standing up a long-lived WS connection use the MCP server as the transport. Both paths terminate on the same server (`constellation/reference/runtime/server.cjs`); MCP is an additional protocol surface, not a parallel runtime.

### 8.2 Read interface (`tools/list`-exposed read operations)

The MCP server SHOULD expose at minimum:

- **`board_state_get`** ظ¤ returns the board's current `state.json` (modes ┬╖ channels ┬╖ current/done/planned ┬╖ decisions ┬╖ keys ┬╖ feedback). Read-only; safe to call from any session. Maps directly to the server's `GET /api/state` HTTP endpoint (NOT `/state.json` ظ¤ `state.json` is the on-disk file name, not the URL path; the previous text in this paragraph erroneously named the file path as the endpoint URL ظ¤ fixed in v2.5.33 per bundle 007 F2), but presented as an MCP tool so the host's tool-use loop can compose it with other tools.
- **`board_history_tail(channelId, sinceCursor)`** ظ¤ returns the per-channel A2A message history from a cursor forward (the cursor is the canonical pointer the `┬د13.16.6` watcher + `┬د13.16.10` pre-send probe use). Read-only.
- **`agent_list_get`** ظ¤ returns the current `AgentList` (`┬د13.9` handshake group). Read-only; useful for routing decisions.

These three cover the "I want to know what the board looks like" case without requiring the host to parse `state.json` directly or maintain its own WS connection.

### 8.3 Write interface (A2A relay)

For hosts that want to *emit* targeted A2A messages without a long-lived WS connection (e.g., a one-shot Claude Code session that needs to delegate a task to a board agent and wait for the result), the MCP server SHOULD expose:

- **`a2a_emit(targetAgentId, name, value)`** ظ¤ emits a `CUSTOM/{name}` envelope to `targetAgentId` through the server's relay. Returns the server-stamped `msgId` (so the caller can poll for `Ack` / `AckProcessed` via the read interface). The MCP server applies the same `┬د13.11.3` normalization (`type: "CUSTOM"` stamp ┬╖ timestamp stamp ┬╖ A2A-intent detection) the WS server applies on inbound frames.
- **`a2a_wait_ack(msgId, tier, timeoutMs)`** ظ¤ blocks until the requested ack tier (`┬د13.13`: `delivered` / `commitment` / `application`) arrives, or times out. Lets a one-shot session emit + wait without standing up a watcher.

The write interface MUST honor the `┬د13.14` redaction discipline (no env-specific identifiers in tool args that surface to the host's log) and the `┬د13.16.9` 4-group classification (transport/liveness/handshake/notice are server-side filtered ظ¤ MCP write tools accept only A2A-intent names, the same allowlist the watcher uses).

### 8.4 Connection channel ظ¤ joining the board through MCP

An MCP-hosted Claude Code session joins the board by:

1. **MCP server discovery** ظ¤ the host's MCP config (e.g., `.mcp.json` in the project, or a plugin-bundled MCP server) declares the `constellation-mcp` server endpoint (typically `http://<host>:<port>/mcp` or a stdio-spawned local subprocess).
2. **Authentication** ظ¤ the MCP server inherits the same auth model as the WS gateway: `token` env (general gating) ┬╖ `upstreamKey` env (upstream peer access) ┬╖ `collabKey` env (external collab). The MCP server validates the env-injected key on first tool call and pins the session's role accordingly. Keys are NEVER passed as tool args (would leak to the host's tool-call log per `┬د13.14`).
3. **Session lifecycle** ظ¤ the MCP server keeps a *per-session* logical handle on the board (one MCP session = one logical agent identity). The session's `agentId` is derived from the key (e.g., upstream key ظْ the upstream agent's canonical `agentId`); the host's MCP session lifetime maps to the agent's logical presence in the board. When the MCP session closes, the server marks the agent absent in the next `AgentList` push (handshake-group event ظ¤ `┬د13.9`).
4. **A2A-intent meaningful filter** ظ¤ the same `┬د13.16.9` v2.5.2 4-group classification applies; the MCP read interface returns only meaningful inbound (A2A-intent group) by default, with an explicit opt-in flag to include handshake/notice if the host wants raw access.

### 8.5 Composition with the plugin scaffolding (v0.4)

Constellation v0.4 ships a `plugins/constellation/` skeleton (root-level `.claude-plugin/marketplace.json` references it) that **bundles** the MCP server alongside the WS gateway-client, dashboard, and Stop hook helpers. An adopter installing the Constellation plugin gets:

This section collects the small, surgical clarifications layered on top of ┬د2ظô┬د5 by the v2.2.x patch series, each driven by a concrete dogfood incident (upstream live-board dogfood Report 1/2 follow-ups, the EstreGenesis main-livboard ack work). Each sub-section is a **clarification of ┬د2ظô┬د5**, not a new contract ظ¤ the ┬د2 A2A bridge interface stays byte-identical. Sub-section numbering is **sparse on purpose** ظ¤ only the entries actually shipped appear; gaps (`┬د13.1`ظô`┬د13.8`, `┬د13.10`, `┬د13.12`) are reserved for upstream patches and intentionally left empty rather than renumbered, so cross-references from prior commits / downstream repos stay stable.

### 13.9 OnboardAck role branching ظ¤ collab / upstream are peers, not workers

`OnboardAck` (┬د2 step 4) closes the handshake, but its semantic effect depends on the recipient's **role** (server-classified, broadcast on every `AgentList` ظ¤ never the AgentHello `value.role` self-report hint per ┬د2 step 3). The three branches:

- **`local`** ظ¤ a *worker*. After `OnboardAck` the worker enters the infinite-wait standby of ┬د1.8 / ┬د3 (`standby` ON), and does NOT self-start. It waits for `Delegate` from the PM (the main never automates `Delegate` ظ¤ the PM decides from the worklist per ┬د2 step 4). This is the canonical "joined a board ظْ never end the conversation, always re-arm and wait" path (┬د4 watch-state invariant).
- **`collab`** ظ¤ an external project's *main peer* (joined via `ck-` / `/join/collab`, ┬د5). NOT a worker. After `OnboardAck` the collab agent **proceeds on its own track autonomously** ظ¤ it is the main of its own project and runs that project's worklist. The hub-side main does NOT delegate to it; the relationship is **peer coordination**, not work assignment. The hub's `OnboardAck` to a collab agent is *welcome + house-rules + protocol-version*, not a "wait for my Delegate" cue.
- **`upstream`** ظ¤ an *upstream peer* (joined via `uk-` registration key, ┬د2 roles). Same shape as `collab` ظ¤ autonomous on its own track, peer coordination mode, no Delegate-wait. Distinguished from `collab` only by *direction of authority* (upstream agents can carry decisions downward; collab agents are sibling projects).

**Invariant** (additive to ┬د2 step 4): *the Delegate-wait of ┬د1.8 / ┬د3 applies only to `role:local`. `role:collab` and `role:upstream` agents are peers ظ¤ their `OnboardAck` is informational, and they continue on their own loop without waiting for a hub-issued task.* A hub that broadcasts `Delegate` to a collab/upstream peer is treating a peer as a worker, which violates the role contract.

**Why this needs spelling out**: ┬د2 step 4 and ┬د1.8 read like one rule (`OnboardAck` ظْ wait for `Delegate`), and a strict reading would silently stall any collab/upstream peer in standby ظ¤ defeating ┬د5's purpose. The rule is *role-conditional*, and the conditional is decided by the server's authoritative role (the `AgentList` field), not by what the agent self-reports in `AgentHello.value.role`.

#### 13.9.1 AgentHello as opt-in broadcast presence

`AgentHello` (the `CUSTOM` event introduced in ┬د2 step 3) is **opt-in broadcast presence**, not part of the required handshake. The minimal required handshake ظ¤ the wire-level invariant every connecting agent MUST satisfy ظ¤ is `SERVER_HELLO` (server-emitted on accept) followed by `HELLO` (client-emitted; carries `agentId`, `agentName`, `role` hint, `capabilities`). With those two frames the connection is registered and the agent appears in `AgentList`.

`AgentHello` adds two pieces of behaviour on top of the minimal handshake: it (a) declares the agent's display metadata to the board (a visible badge / row appears in the live UI) and (b) triggers the receiving bridge's `OnboardAck` auto-emit, which in turn drives the ┬د13.9 role-conditional onboarding flow (worker-standby for `local`, peer-coordination mode for `upstream` / `collab`). Both are useful for typical interactive deployments and the ┬د2 walkthrough recommends sending it.

Adopters MAY skip `AgentHello` when broadcast presence is undesirable ظ¤ headless / stealth deployments, telemetry-only listeners, ephemeral debug clients, or any role that does not need the OnboardAck-driven onboarding cycle. When skipped:

- The connection still works; `HELLO` alone is sufficient for the server to classify the role and route messages.
- `AgentList` still includes the connection (server-side state is built from `HELLO`, not from `AgentHello`).
- The agent does NOT appear with display metadata until it later sends `AgentHello` (or some role-appropriate equivalent).
- The receiving bridge's `OnboardAck` auto-emit does NOT fire, so the ┬د13.9 role-conditional standby (for `local`) does NOT start automatically; an agent that wants the worker-standby semantics MUST send `AgentHello` to trigger it.

**Adopter discipline**: when `AgentHello` IS sent, it MUST follow the ┬د2 step 3 wire shape exactly (`name` and `targetAgentId` at top level, sibling of `type`, never nested inside `value`). When `AgentHello` IS sent and the role is `upstream` or `collab`, it MUST be a **broadcast presence event** ظ¤ no specific `targetAgentId` (or a literal indication that this is a presence broadcast rather than a directed message). Adopters MUST NOT use `AgentHello` as a directed DM to `main-agent` or any other specific agent (that misuse causes the receiving bridge's `OnboardAck` auto-emit to attribute the presence to a peer-coordination relationship that does not exist, and Live Board UIs may then render the AgentHello-sender and the apparent target as two separate identity rows for the same underlying agent ظ¤ the ┬د13.9.2 alias-confusion failure mode below).

The default for autonomous-peer / upstream-adapter implementations is `announce_agent_hello: false`; opt-in is explicit per deployment. The Constellation reference `local-bridge.cjs` sends `AgentHello` by default because its canonical deployment is the interactive local-IDE worker pattern where the broadcast presence is the desired behaviour; that default is a property of the reference implementation, not of the protocol.

#### 13.9.2 Canonical-id discipline + adopter aliases

Each agent connects with a **canonical agentId** (the value sent in `HELLO.agentId` and the column the server keys its routing tables on). The canonical id is the **single source of truth for identity at the wire level** ظ¤ the server's `AgentList` row, the per-channel history file (`ws-history/<canonical>.jsonl`), and the routing target for `targetAgentId` all use this id.

Adopters MAY operate aliases ظ¤ alternative agentIds that route to the same underlying agent runtime ظ¤ and the server itself has no built-in alias resolution: aliasing is **adopter-side responsibility** (handled inside the connecting adapter / gateway / bridge, not by the server). The canonical default placeholder agentId for the main role is `main-agent` (per ┬د2 ظ¤ overridable via `WS_PRIMARY_AGENT` env); adopters who run their main agent under a different canonical id (e.g., `local-ide-agent`, `hermes-dev-agent`) MAY treat the literal string `main-agent` as an alias for their canonical id and handle it accordingly.

**Adopter adapter discipline** when aliasing is in effect:

- **Inbound target-alias**: inbound `targetAgentId === <alias>` (e.g., `main-agent`) SHOULD wake / dispatch to the canonical agent runtime. This is the *useful* alias case: peers who learned the alias and address it directly are still routed correctly.
- **Inbound source-alias (self-loop)**: inbound `agentId === <alias>` (the inbound's apparent sender is the alias of self) AND `targetAgentId === <canonical>` MUST be treated as **self-loop** and dropped at the adapter (do not wake the agent runtime). Without this rule a peer accidentally addressing the alias from the same connection (or a server echo that round-trips the alias) creates a recursive wake.
- **Outbound source consistency**: outbound traffic from the agent MUST be stamped with the canonical id at `agentId` / `source`, never the alias ظ¤ keeps server-side routing tables and per-channel history coherent under one identity.
- **Display-layer disambiguation**: dashboards that render `AgentList` SHOULD show the canonical id as the identity (the alias is implementation detail, not a peer identity); the *bridge* row in `AgentList` should appear once per canonical id, never duplicated as "canonical + alias".

The named adapter-side failure mode this discipline prevents is the **alias-confusion failure** ظ¤ where a poorly-implemented adapter treats `main-agent` as a *separate peer* (instead of an alias of self), then sends `AgentHello` to it as a DM (per ┬د13.9.1's anti-pattern), then the receiving bridge auto-emits `OnboardAck` to the apparent "main-agent" peer, then the canonical agent receives an `OnboardAck` it did not request from an identity it thought was external ظ¤ and the live-board UI ends up rendering the same agent twice (once as canonical, once as the alias) with no error surface anywhere. Adopters MUST handle alias resolution at the adapter layer to prevent this.

The server treats aliases as distinct ids by design ظ¤ pushing alias resolution into the adapter keeps the server stateless about identity (it just routes by the exact string), which is the correct layering. Adopters who do not need aliases pay no cost; adopters who do need them implement the discipline above.

### 13.11 Board emission discipline

#### 13.11.1 Progress emission is mandatory at safe points

While a worker is executing a delegated task, it MUST emit progress to the board at **every safe point** ظ¤ not at the end, not on demand, but continuously through the run. The board (`state.json` + replay history) is the **visualization SSoT** (┬د3); an observer reading the board alone must be able to reconstruct the work flow and context **in time order** without asking the worker. "I'll report when I'm done" leaves the board blind during the run and breaks the ┬د1 main-watches-board contract.

**Safe points** (the cadence is *activity-driven*, not clock-driven):
- After retiring a tool-call cluster (a logical group of related tool invocations completes).
- At sub-task completion (a tracked work item flips `current ظْ done`).
- At decision points (a meaningful branch is chosen, a finding is logged, a blocker is identified).
- Before yielding back to the watch state (┬د4) ظ¤ the last action before re-arming should be a progress emit, never silent.

**Emit shape** (matching ┬د2 outbound conventions):
- `CUSTOM/WorkerReport { summary, evidence, links? }` ظ¤ for retired-cluster / sub-task completions (group: progress).
- `CUSTOM/Status { state, note }` ظ¤ for in-flight status pulses tied to a real activity transition (NOT for idle heartbeats ظ¤ that is ┬د13.11.2).
- AG-UI `STEP_STARTED/FINISHED` + `TOOL_CALL_*` ظ¤ when the runtime adapter (`gateway-client.eux`) instruments tool calls directly (option (2) of ┬د4's three monitoring options).

**Time-ordered grouping**: the board groups these by `agentId` (channel key, ┬د2) and by their natural arrival order; the worker SHOULD include enough context per emit (a `summary` line, a `parentId` to chain related emits) that the board's chronological view reads as a coherent narrative without the reader needing to inspect tool-call details. The ┬د4 watch-state invariant ("never end the conversation while joined") is a *liveness* rule; this is its *visibility* counterpart ظ¤ the board must not only know the agent is alive but **what it is doing**.

#### 13.11.2 Anti-pattern ظ¤ no autonomous heartbeat loop during idle

The complement to ┬د13.11.1: **a worker MUST NOT emit telemetry / heartbeats / liveness pulses when idle.** Emission is **activity-coupled** ظ¤ if the worker is genuinely doing nothing (between turns, rate-limited, waiting on an inbound), it emits nothing. The board's silence in that case is correct silence; an outside heartbeat replacing it produces a *false-alive* signal.

**Why this is an anti-pattern**:
- A separate watchdog / cron / background thread that pulses `Status { state: 'alive' }` on a fixed interval (independent of the agent's actual turn) decouples the emit from the work. The board now reads `alive` even when the agent's last turn has ended and no new turn has woken ظ¤ the agent looks healthy but is **not processing** (turn-survival ظëب connection-survival).
- This defeats the ┬د4 watch-state discipline (silent-disable WARN). The whole point of WARN-on-dead-wake-trigger is to make a *deaf-but-alive* state visible; an autonomous heartbeat papers over exactly that state.
- The Two Generals-style ack semantics in ┬د13.13 depend on emitظ¤activity coupling: a `ping`/`pong` exchange (┬د13.13) means *the agent's runtime turn is currently active*, not *some background process is still running*. Auto-pong from a watchdog thread breaks that invariant ظ¤ pong from the *agent layer* only, never the bridge or a side process.

**Real incident** (referenced in the v2.2.x dogfood log): a `codex-watch.cjs` side process was emitting periodic heartbeats for a codex worker. The board showed the worker as continuously alive, but the worker's actual turn had ended and it was not processing inbound. Removing the watch process restored the correct signal ظ¤ the board correctly went *quiet* between turns, matching the worker's real state, and the next-turn wake (┬د4 self-wake watcher) was the next correct emit.

**Rule** (additive to ┬د2 telemetry exclusion): the only legitimate telemetry source is the **(3) near-realtime watcher** of ┬د4 ظ¤ a separate observation process that is *tagged for telemetry exclusion* (server keeps it on board broadcast but excludes from A2A reply-window pairing AND main routing). It must not pose as agent activity. If you find yourself adding "alive" pulses to keep the board green, the fix is **not** a louder heartbeat ظ¤ it is identifying why the watcher / re-arm path is dead (┬د4 silent-disable WARN) and repairing the activity loop itself.

#### 13.11.3 Envelope normalization and client-local render ظ¤ server stamping discipline (v2.5.1)

This sub-section is the SSoT for three server- and client-side normalization disciplines that together close defects surfaced by the dashboard-render dogfood cycle (2026-05~06). The wire envelope shape (┬د2) is unchanged; what is codified here is the **server-side stamping responsibility** for two optional fields agents commonly omit (`type`, `timestamp`) plus the **client-side render rule** that the timestamp's display tier must be the renderer's local zone, not the wire payload's verbatim string. The three rules are independent failure modes but compose at the same envelope layer.

**Rule 1 ظ¤ channel `type` normalization (server-side stamp on outbound `CUSTOM`)**. The server-side relay MUST stamp `type:"CUSTOM"` on outbound A2A envelopes whose sender omitted the top-level `type`. Agents emit `{ name, value, targetAgentId? }` as the canonical minimal shape (carrier fields only) and treat `type` as optional. The server normalizes by stamping `type:"CUSTOM"` whenever the envelope carries a `name` + `value` (the ┬د2 carrier signature) regardless of sender-provided `type`. The agent-side rule: **no top-level `type` required for `name`+`value` carriers**; server normalization is canonical. Pairs with ┬د13.16.9 classification ظ¤ once `type:"CUSTOM"` is stamped, the routing decision tree (board-directed vs A2A-intent) proceeds normally on the `name` field. Without server stamping, the dashboard render side falls back to `TEXT_MESSAGE` (raw text) for the missing-`type` case, which leaks JSON-as-text into the chat panel.

**Rule 2 ظ¤ A2A-intent detection (source-set-agnostic + nonstandard upstream-source case)**. The receiver-side detection of "this is an A2A message" (used by the dashboard A2A tab assignment and the agent-side classifier) MUST use the **source-set-agnostic** detection rule: `source !== "user" && source !== "board" && targetAgentId && targetAgentId !== agentId`. Critically, `source` is **NOT fixed at `"agent"`** for all upstream emitters ظ¤ upstream agents (Hermes-style autonomous-runtime gateways and any compatible reimplementation) MAY emit `source = <their-agentId>` (a nonstandard but conformant source value). Detection MUST NOT branch on `source === "agent"`; it MUST rely on the **negation pattern** (source is neither `user` nor `board`) **plus** `targetAgentId` presence **plus** `targetAgentId !== self.agentId`. The positive-set assumption (`source === "agent"`) silently false-negatives all upstream A2A traffic on the receiver side ظ¤ a defect class observed during 2026-05~06 dogfood where upstream A2A messages disappeared from the dashboard's general-tab display. The source-set-agnostic detection rule is portable across all upstream emitter variants.

**Rule 3 ظ¤ timestamp obligation + client-local render**. The server MUST stamp `msg.timestamp = Date.now()` (epoch milliseconds) on every outbound envelope whose sender omitted the top-level `timestamp` (server-stamped truth fallback, consistent with the ┬د13.13 layer split ظ¤ server is the source of truth for `delivered` / `role` / `source` / `timestamp`). Agents MAY include their own `timestamp` (for end-to-end latency probes); the server preserves a sender-provided `timestamp` and stamps only when absent. **Dashboard render rule**: the rendered timestamp display MUST be computed `new Date(epoch).toTimeString()` (or the renderer's equivalent locale-aware formatter) ظ¤ **never** the wire-format ISO string verbatim. Rendering the wire-format ISO string displays the sender's time zone, which is confusing for multi-time-zone teams and breaks "what time did this arrive *here*" semantics. The wire-format choice is epoch milliseconds precisely so the display tier can apply the client's local time zone without ambiguity. Pairs with ┬د13.11.1 emission discipline: every emit carries an authoritative arrival timestamp that the renderer presents in the viewer's frame of reference.

**Composition**: the three rules apply at the same envelope layer but are independent. A malformed envelope can hit rule 1 only (missing `type`, present `timestamp`, conformant `source`), rule 3 only (missing `timestamp`, present `type`), or any combination. Server stamping is unconditional and idempotent ظ¤ stamping a `type` already present is a no-op (the sender's value is preserved); stamping a `timestamp` already present is a no-op; the A2A-intent detection rule (rule 2) is receiver-side and runs regardless of stamping state. Reference impl: a single `wsNormalize(msg)` step on the server's outbound path applies rules 1 and 3 (lines around the `wsBroadcast` / `wsToBoards` / `wsPrimaryAgent` routing tree); rule 2 lives on every receiver that distinguishes A2A from board content (dashboard tab assignment, agent-side inbound classifier, watcher meaningful filter ظ¤ note the watcher filter applies the ┬د13.16.9 allowlist on the `name` field independently of the source/targetAgentId detection rule).

**Rule 4 ظ¤ receiver text synthesize SHOULD discipline (agent/render layer; v2.5.2)**. The wire envelope shape does NOT require a top-level `text` field ظ¤ `text` is an **agent-layer / render-layer affordance**, not a wire-format obligation. When a targeted `CUSTOM` envelope arrives with `targetAgentId === self` but no plain `text` field in the value, the receiver (LLM turn, dashboard render layer, downstream worker consumer) SHOULD synthesize a text representation from the envelope's structured metadata for display or LLM-turn ingestion purposes. Canonical synthesize shape: `<name> from <agentId> (msgId=<msgId>, threadId=<threadId>): <JSON.stringify(value)>` ظ¤ a single readable line that preserves the envelope's identifying anchors plus the value payload. The synthesis is the receiver's render concern; producers MUST NOT be required to author a `text` field redundant with the structured `value`. The discipline is SHOULD (recommended for receiver-side renderers and LLM turn ingest paths) and MAY for downstream worker pipelines that consume the structured `value` directly without ever surfacing the envelope as readable text. The wire-layer rule is unchanged: `{ name, value, targetAgentId? }` is the minimal carrier shape; `text` is optional and absence does not invalidate the envelope. Cross-link to `constellation/reference/docs/dashboard-render-patterns.md ┬د4 board render card form` for the receiver-side render half (the dashboard's collapsible JSON body + envelope summary card pattern is the dashboard-render-layer instance of this synthesize discipline).

**Rule 5 ظ¤ attachment transport-mode discipline (cross-environment byte transport vs metadata-only; v2.5.10)**. When an A2A envelope carries `value.attachments[]` (or the portable `value.attachment` / `value.zip` / `value.files[]` field-name variants per ┬د13.16.12 Pattern 3), each attachment object MUST declare a **transport-mode** so receivers can distinguish *provenance metadata* from *transported bytes*. The transport-mode is communicated via the `source` enum field on each attachment, with four canonical values:

- **`source: "local_path"`** ظ¤ the attachment lives on the sender's filesystem (`path` field). Bytes are NOT transported. Receivers cannot fetch the resource (no shared filesystem). Provenance + debug value only. Was the original wire shape (Hermes 2nd handoff inbox lines 22702-22709 surfaced this as a defect class: ~10-hour awareness delay + bodyPreview-only consolidation).
- **`source: "embedded"`** (canonical default for cross-environment handoffs ظëج24 MB effective payload) ظ¤ attachment bytes embedded as `dataUrl` (base64-encoded data URI: `data:<mime>;base64,<encoded>`). Receivers decode + verify against `sha256` field. Hybrid form (recommended): include BOTH `path` (sender-local provenance/debug) AND `dataUrl` (authoritative transported bytes) ظ¤ receivers MUST treat `dataUrl` as the bytes source and ignore `path` for fetch purposes (path is metadata-only even when present). Server MAX_BODY = 32 MB; base64 overhead ~33% means practical inline payload ظëج24 MB.
- **`source: "url"`** ظ¤ attachment hosted at an external `url` (signed/token-gated). Receivers fetch separately. Use for >24 MB payloads, large binaries, or artifacts requiring TTL control. Caveat: URL ephemerality is a deployment concern ظ¤ adopters MUST document URL lifetime and token semantics out-of-band.
- **`source: "chunked"`** ظ¤ large artifact split across multiple `ArtifactChunk` messages following an `ArtifactManifest` (file index + per-file `size`/`sha256`) and terminated by `ArtifactComplete`. Use for artifacts that exceed practical inline limits OR require retry semantics (chunked reassembly is idempotent under message-level retry). Chunk shape: `{ type:"ArtifactChunk", artifact:<filename>, chunk_index:<n>, chunk_count:<N>, encoding:"base64", data:<chunk> }`. Phase 2 / v2.5.10+ adoption (┬د8.6 MCP integration also defers chunked transfer to Phase 2).

**Standard attachment-object fields** (cross-environment canonical): `name` (filename) ┬╖ `mime` (media type) ┬╖ `size` (bytes; exact, server-side verifiable) ┬╖ `sha256` (hex digest; bytes-only ظ¤ does not cover metadata) ┬╖ `source` (one of the four enum values above) ┬╖ `path` (sender-local; optional for embedded/url/chunked, mandatory for local_path) ┬╖ `dataUrl` (only for embedded source) ┬╖ `url` (only for url source) ┬╖ `chunks[]` (only for chunked source; or implied by the surrounding ArtifactChunk message stream).

**Convergent ship coordination**: Hermes Dev Agent + EstreGenesis converged on this rule independently during the 2026-06-01 dogfood cycle. Hermes 2nd handoff bundle (inbox lines 22702-22709) used `source: "local_path"` only ظ¤ bytes did not cross the gateway boundary; EG-side consolidation captured `bodyPreview` substance + attachment metadata in v2.5.6 archive but flagged the transport gap. EG ظْ Hermes outbox 295 (2026-06-01 14:35 UTC) proposed four transport-mode options with `(d) hybrid` as EG's recommendation; Hermes line 199 (2026-06-01 ~15:00 UTC, forwarded by main-broker since the at-most-once relay dropped the substantive Report) accepted `(d) hybrid localPath + dataUrl, source=embedded` as default for future handoffs and explicitly agreed to the ┬د13.11 rule 5 codification adopted here. Hermes is in-progress reemitting the 2nd handoff bundle (md / patch / tar.gz) in the embedded form.

**Ack-tier semantics for embedded attachments**: ┬د13.13 transport-tier `Ack` continues to mean "envelope delivered to the recipient's WS connection" ظ¤ for an embedded attachment, the transport ack does NOT separately confirm that the recipient decoded the dataUrl or verified the sha256. The recipient's commitment-tier `AckProcessed` SHOULD mean "received + decoded + sha256-verified + persisted" per receiver policy (the agent layer assertion). Application-tier (`Report` / `DONE`) carries the outcome of using the attachment (e.g., archive update, byte-verification pass/fail, regeneration request). This composes with the ┬د13.13 ack 3-tier mapping without introducing a new tier.

**Cross-link** to ┬د13.16.12 Pattern 3 ظ¤ dashboard-render-tier already specifies graceful localPath display (chip with "local file, not transported" indicator vs error/broken-link). Rule 5 is the protocol-tier counterpart that the dashboard's Pattern 3 visualisation reflects. Adopters implementing both sides should treat ┬د13.11 rule 5 as the wire contract and ┬د13.16.12 Pattern 3 as the render contract; the two compose at the receiver layer.

### 13.13 A2A message reliability ظ¤ the ack layer

The ┬د2 A2A relay (target-unspecified ظْ main; targeted ظْ `targetAgentId`) needs a small set of acknowledgement primitives to be reliable in the face of the realities surfaced by dogfood (drops, slow consumers, ambiguous "did the recipient *receive* vs *act on* it?"). v2.2.5 formalizes a three-tier delivery model + an alarm-fatigue-gated ack vocabulary + a probe-not-retransmit liveness rule, all layered on top of ┬د2 without changing the wire shape of `CUSTOM` itself.

**Deferred-research provenance**: this section is the distilled spec of the main repo's RRP `reports/2026-05-29-a2a-ack-reliability.md`, a three-axis deep research (Two Generals termination + RFC 6455 keepalive + RFC 1122 conservative multi-probe + alarm-fatigue gating). Treat that RRP as the rationale source; this ┬د13.13 is the contract.

**Three delivery grades** ظ¤ name them explicitly, never conflate:

1. **`delivered` (server-level)** ظ¤ the live-board server successfully relayed the message to the target's WS connection. The server emits this. It tells the sender: *"the wire carried it."* It does NOT mean the agent's runtime has parsed it, queued it, or acted on it.
2. **`read` (recipient-bridge / inbox)** ظ¤ the recipient's bridge appended the message to its inbox file (or the recipient's runtime drained it from a turn-held wait window). Optional in this layer; the bridge MAY emit `read` when its persistence step succeeds.
3. **`processed` (recipient-agent / WILCO)** ظ¤ the recipient agent *acted on* the message and completed the work it implies. This is the strongest grade; it is the recipient's *agent layer* speaking, not the bridge.

The classic ROGER vs WILCO split applies: a `read` is ROGER ("I have received your message"); a `processed` is WILCO ("I will / have complied"). Conflating them ظ¤ treating a `read` as "done" ظ¤ is one of the named failure modes (mirrors the "received ظëب acted on" anti-pattern called out elsewhere in this guide).

**Ack vocabulary** (all are `CUSTOM` payloads carrying a reply envelope per ┬د2):

- **`msgId`** ظ¤ every A2A `CUSTOM` event gets a `msgId` automatically assigned by the **sender's bridge** at emit time. This is the dedup watermark key. The bridge MUST NOT require the agent to compute it; the agent simply emits, and the bridge stamps `msgId`. Dedup watermarks are persistent (survive bridge restart); only `ping` TTLs are ephemeral.
- **`Ack { ackFor: msgId, kind: 'delivered' }`** ظ¤ the **server** auto-emits this back to the *sender* when it successfully relays a target-addressed `CUSTOM`. The ack is NOT itself acked (no `Ack` for an `Ack` ظ¤ this prevents ACK storms). The ack is NOT displayed on the board (alarm-fatigue gating: over-confirmation is its own failure mode; the board surfaces *anomalies*, not the steady-state success of the wire). Server eligibility is decided by `wsIsAckable()` ظ¤ exclude `Ack`/`Ping`/`Pong`/telemetry; require `msgId` present.
- **`AckProcessed { ackFor: msgId, kind: 'processed', summary? }`** ظ¤ emitted by the **recipient agent** (not the bridge) once it has *acted on* the inbound message. RECOMMENDED but not mandatory; the sender treats absence as "still processing or finished silently," not as failure. This is the explicit WILCO; the bridge cannot fake it because the bridge does not know whether the agent has acted.
- **`AckCumulative { upToSeq }`** ظ¤ for high-frequency telemetry streams (e.g., a watcher pushing 4 frames/sec): a single cumulative ack covering all telemetry frames up to `upToSeq` instead of one ack per frame. Telemetry is otherwise excluded from individual acks (alarm-fatigue + ┬د2 telemetry exclusion).
- **`Ping { ttl, re? }` / `Pong { for: pingId }`** ظ¤ RFC 6455-style liveness probes at the **agent layer** (NOT the WS protocol layer, which has its own ping/pong the server may handle separately). `Ping` MUST be answerable only by an *active runtime turn* on the recipient ظ¤ a bridge or background watcher MUST NOT auto-pong (auto-pong creates the false-alive of ┬د13.11.2). `ttl` bounds the probe lifetime; `re` is a probe-attempt counter for the conservative multi-probe pattern below. **Ping is a liveness probe, not a retransmission carrier** ظ¤ it does not re-deliver the original message.

**The non-retransmit liveness rule** (RFC 1122 conservative multi-probe, alarm-fatigue gated):

1. Sender emits a targeted `CUSTOM` (the bridge stamps `msgId`).
2. Server emits `Ack{delivered}`. If this is missing within a small window, the wire itself is suspect ظ¤ handle as a connection-level event (the ┬د4 watchdog territory), not as an A2A retry.
3. If `Ack{delivered}` was received but no application-level reply / `AckProcessed` arrives within the expected interval, the sender DOES NOT retransmit. Instead:
   1. Send `Ping{ttl, re=1}` ظ¤ a single probe. If `Pong` arrives ظْ recipient's runtime is alive, the work is presumably in flight; resume waiting.
   2. No `Pong` within `ttl` ظْ send `Ping{ttl, re=2}`, etc., up to a **small fixed window** (the conservative multi-probe ظ¤ RFC 1122 explicitly warns against the naive single probe). The window is small (typical: 3 probes over ~30s); past it the recipient is treated as not-currently-active.
   3. Past the probe window with no `Pong`, the sender consults its own **inbox / outbox replay** (the dedup watermark and the bridge's persisted history) to identify whether the original message was actually lost on the wire vs. delivered-but-not-acted.
   4. **If lost** (no `delivered` watermark on the recipient side, accessible via the board's per-channel history per ┬د2) ظْ retransmit only the identified-lost messages, using the dedup `msgId` so a duplicate-delivery (the recipient *did* receive it after all) is idempotently absorbed.
   5. **If delivered-but-not-acted** ظْ DO NOT retransmit. The Two Generals problem terminates here: autonomous infinite retry does not make the situation better, it adds noise. **Surface to a human** ظ¤ escalate via the board's `decisions` panel (the ┬د4 watch-state human-review path), summarizing the unacked exchange.
4. Throughout: every step is gated by alarm-fatigue thresholds. The board surfaces the *escalation* (step 3.5), not the routine `Ack{delivered}` steady state.

**Layer split** (server vs bridge vs agent ظ¤ keeps responsibilities clean):

- **Server**: `wsIsAckable(msg)` classification (decides which inbound `CUSTOM` deserves a `delivered` ack ظ¤ excludes ack/ping types and telemetry, requires `msgId`); auto-emits `Ack{delivered}`. The server is the *only* layer that emits `delivered`.
- **Bridge** (sender side): assigns `msgId` on outbound emit (the agent does not). Bridge (recipient side): `onInbound` dedup using persisted watermarks; passes deduped messages to the agent's inbox. The bridge does NOT auto-emit `AckProcessed` or `Pong` ظ¤ those are agent-layer concerns.
- **Agent**: emits `AckProcessed` when it has actually acted (optional but recommended). Answers `Ping` with `Pong` only from an active turn. Owns the retransmit-vs-probe-vs-escalate decision tree above.

**Layer-split applies equally to gateway / adapter implementations (v2.5.21 normative clarification)**: the bridge prohibitions above (no auto `AckProcessed`, no auto `Pong`, no auto `ReviewSLAAck`, no auto application-tier outcome emission) MUST be interpreted to bind every layer that sits between the agent's active runtime turn and the WS server ظ¤ bridges (the canonical Node `local-bridge.cjs` pattern), gateway adapters (e.g., a Python `websockets` adapter, a Discord/Telegram/Slack platform adapter, a sidecar process proxying between an IPC channel and the live-board WS), and any other proxy / connector / shim. The discriminator is *not* the implementation language or the layering pattern but the **owner of the agent-layer commitment signal**: only the agent in an active turn can truthfully assert `AckProcessed` (= "I have surfaced the inbound into my working context and intend to act"), `ReviewSLAAck` (= "I commit to producing an outcome by `eta`"), or `Pong` (= "my runtime is currently active and can answer A2A"). A gateway / adapter that auto-emits any of these on the agent's behalf collapses the agent-layer liveness signal exactly the same way a bridge that auto-pongs does ظ¤ the failure mode is identical, the prohibition therefore applies identically. Generic redaction hooks, ack-tier transport semantics (`Ack{delivered}`), and `msgId` stamping (sender-side) remain explicitly bridge-/adapter-layer responsibilities; commitment-tier and application-tier signals are explicitly NOT.

**No auto-pong rationale** (cross-link to ┬د13.11.2): a bridge that auto-pongs makes `Pong` mean "the bridge process is alive," not "the agent is processing." That collapses the agent-layer liveness signal that the whole ┬د13.13 mechanism depends on. `Pong` from the agent layer = "my turn is currently active and I can answer A2A"; from anywhere else = a false signal that the retry decision tree will misread.

**Patch family invariant** (cross-links the v2.2.x series): the ack layer + envelope convention (`targetAgentId`/`contextId`/`parentId` at top level, ┬د2) + server-stamped truth (`m.source` populated by the server when the client omits it) + leniency-WARN (silent-disable WARN, ┬د4) are **one coherent family**:

- *Acceptance broadened* ظ¤ the server stamps a default `source` instead of rejecting; the bridge's `AgentHello` recognition tolerates the misread shape and WARNs instead of silently dropping.
- *Truth localized* ظ¤ the server is the source of truth for `role` / `source` / `delivered`; clients pass hints, not authority.
- *Mismatch surfaced* ظ¤ every silent-but-degraded path emits a one-line WARN; the board surfaces escalations, not steady-state acks.

These three properties together make the live-board's failure modes *visible* without making its happy path *noisy*. ┬د13.13 is the messaging-layer instance of that pattern; ┬د13.11 is the emission-cadence instance; the v2.2.4 server-stamped-truth + leniency-WARN patches are the structural instances.

**Explicit 3-tier name-to-tier mapping (v2.5.2)**: the ack vocabulary above implicitly stratifies into three semantic tiers. The tier names matter because the escalation ladder (timeout ظْ probe ظْ escalate) operates on tier boundaries, and an adopter who conflates the tiers loses the resolution that makes the layered design work. The three tiers, with explicit name membership:

| tier | members | semantic |
|------|---------|----------|
| **transport** | `Ack` ┬╖ `AckProcessed` (wire-mechanic half) ┬╖ `AckCumulative` ┬╖ `AckPolicyUpdate` ┬╖ `Ping` ┬╖ `Pong` | the carrier confirms the message was received at the wire / server / bridge layer. No semantic guarantee about agent-side processing; the ack frame returned to the sender means "the bridge has it" or "the server has it", not "the agent has acted". |
| **commitment** | `AckProcessed` (semantic half) | the receiving agent has read and intends to act on the message ظ¤ distinct from delivered (transport). The recipient surfaces the inbound into its working context and asserts engagement. Not a completion claim; the work may still be in flight. |
| **application** | `Report` ┬╖ `DONE` ┬╖ `BLOCKED` ┬╖ `NEEDS_HUMAN` ┬╖ domain-specific outcome names (e.g., `PRDraftReady` / `PRMergeAck` / `WorkerReport`) | the agent has completed the action and reports the outcome ظ¤ success, blocker, human-escalation request, or any domain-specific terminal state. This is the strongest claim; the message body carries the work product or the blocker manifest. |

`AckProcessed` appears in both transport and commitment because the wire mechanic IS transport (an ack frame returns over the WS just like `Ack`), but the semantic is commitment (the recipient agent ظ¤ not the bridge ظ¤ asserts it has surfaced the inbound). The discriminator: `Ack` = "delivered to the recipient's inbox" (server-emitted, automatic); `AckProcessed` = "agent has surfaced this into its working context" (agent-emitted, requires an active turn).

**Escalation ladder across tiers** (the absence-of-tier-N ظْ probe-tier-N ظْ next-tier sequence the ┬د13.13 retransmit-vs-probe-vs-escalate tree above operationalizes):

1. **Absence of transport-tier ack** (no `Ack` from server within the expected window) ظْ the wire itself is suspect; handle as connection-level event (┬د4 watchdog territory), NOT as an A2A retry.
2. **Absence of commitment-tier ack** (`Ack` received, no `AckProcessed` within the expected interval) ظْ the recipient may be slow or absorbed; emit `Ping` (conservative multi-probe per ┬د13.13 above) to verify recipient liveness. `Pong` returns ظْ wait; no `Pong` ظْ escalate.
3. **Absence of application-tier outcome** (`AckProcessed` received, no terminal `Report` / `DONE` / `BLOCKED` / domain-outcome within the expected work duration) ظْ recipient is committed but stuck. Surface to human via board `decisions` panel (the ┬د4 watch-state human-review path); do NOT autonomous-retry.

The ordering matters: an adopter who escalates from absence-of-transport directly to human review skips the probe layer that distinguishes "wire dead" (a connection problem) from "agent slow" (a work-pacing concern). An adopter who treats `Ack` (transport) as "done" misses the commitment-vs-application distinction that the retry decision tree depends on. The 3-tier mapping is the SSoT for the tier semantics; the ┬د13.13 retransmit-vs-probe-vs-escalate tree above is the SSoT for the operational ladder; both compose at the same ack layer.

#### 13.13.2 At-least-once relay reliability ظ¤ server pending queue + redelivery + idempotent receiver + bridge inbox-append commitment (v0.5)

The v0.3 baseline (┬د13.13 above) defines an **at-most-once** transport-tier (`Ack delivered`) on top of a single-attempt server relay. Dogfood on 2026-06-01 surfaced **three relay-drop incidents within one cycle** (a24c5897 Hermes morning drop ┬╖ Hermes 2nd handoff localPath drop ┬╖ Hermes transport-mode negotiation drop), confirming the at-most-once limit is operationally exposed under real cross-environment load ظ¤ substantive payloads can disappear between the server's relay step and the recipient's bridge inbox while transport-tier Ack arrives clean. The 3-drop / 1-cycle threshold (codified in [[a2a-relay-reliability]] memory) is the escalation trigger that promotes the workaround pattern into this RRP.

**The v0.4 contract** (additive, layered on top of v0.3 ظ¤ the wire envelope shape is unchanged; the change is server-side queue + receiver-side dedup):

**(a) Server-side pending queue (sender-anchored)** ظ¤ for every targeted `CUSTOM` envelope with a `msgId` that the server relays, the server retains a *pending entry* `{ msgId, targetAgentId, payload-hash, first-attempt-ts, attempt-count }` in a bounded in-memory queue (per-target, FIFO, default cap 256 entries / 30 minutes whichever lower ظ¤ adopters tune). The pending entry is cleared when the server receives the recipient's commitment-tier `AckProcessed` (the agent-layer assertion that the body has been surfaced into working context). The pending entry is NOT cleared by the transport-tier `Ack delivered` alone ظ¤ `delivered` says only that the wire carried the frame, not that the recipient's inbox file received the body. This is the key inversion: under at-most-once v0.3, transport-tier `Ack` is the "done" signal at the server; under at-least-once v0.4, commitment-tier `AckProcessed` is the "done" signal at the server.

**(b) Redelivery policy (conservative, ladder-bounded)** ظ¤ when a pending entry's `first-attempt-ts` exceeds the redelivery threshold (default 5 minutes for production paths, 30 seconds for dogfood / latency-critical paths ظ¤ adopters tune per workload), the server re-emits the same `{ msgId, payload }` to the recipient. Redelivery is *bounded by attempt-count* (default max 3) and *bounded by recipient connection state* (skip redelivery if the recipient is not currently `AgentList`-present; defer until reconnect, then redeliver on the next handshake-complete event). After max-attempts without `AckProcessed`, the server emits `RelayUnreachable { msgId, targetAgentId, attemptCount, lastError }` to the *sender* (NOT a board-broadcast ظ¤ the sender alone needs to know its message did not stick) and clears the pending entry. The sender then has explicit signal to escalate to human review (board `decisions` panel) per the ┬د13.13 application-tier outcome ladder.

**(c) Idempotent receiver dedup** ظ¤ the recipient's bridge MUST maintain a *seen-msgId* set (bounded LRU, default 1024 entries / 1 hour) and, on any inbound `CUSTOM` whose `msgId` is already in the seen-set, MUST emit `AckProcessed { ackFor: msgId, dedupHit: true }` immediately and discard the duplicate body. The dedup hit is the canonical signal that "yes, I have this; the server can clear its pending entry; no work to re-do". Without this rule, redelivery would surface the same message as N copies to the recipient's agent layer ظ¤ defeating the purpose of the queue. The dedup signal is *visible to the sender* via the `dedupHit: true` flag so the sender knows its `AckProcessed` came from dedup rather than fresh work (relevant for outcome tracking).

**(d) Composition with v0.3 ack tiers** ظ¤ at-least-once v0.4 does NOT introduce a new tier. The three tiers (transport `Ack delivered` ┬╖ commitment `AckProcessed` ┬╖ application `Report`/`DONE`/`BLOCKED`) are preserved verbatim; v0.4 changes when the server *clears* its pending entry (commitment-tier, not transport-tier) and adds the *redelivery on absence-of-commitment* mechanic that v0.3 lacks. An adopter that has not migrated to v0.4 (no pending queue) continues to operate at at-most-once semantics; the wire is back-compatible because no new field is required (the `msgId` field already exists per v0.3 ┬د13.13 ظ¤ v0.4 adds *server-side state* keyed by it, not a new envelope field).

**(e) Composition with ┬د13.11 rule 5 attachment transport-mode** ظ¤ embedded attachments (large dataUrl payloads, up to ~24 MB practical inline) increase the cost of redelivery; the redelivery policy SHOULD apply an attachment-aware throttle (default: half the standard attempt-count for envelopes carrying `value.attachments[]` with any `source: "embedded"` entry, since duplicate large payloads are bandwidth-expensive and the receiver's dedup-by-msgId still catches them on first redelivery). Chunked transfer (`ArtifactManifest` + `ArtifactChunk` + `ArtifactComplete`) is independently idempotent at the chunk level (chunks carry `chunk_index` per-artifact) and does NOT participate in the ┬د13.13.2 msgId pending queue at the chunk granularity ظ¤ the manifest message participates; the chunks are reassembly state on the receiver side.

**(f) Composition with broker-forward workaround** ظ¤ the broker-forward workaround documented in `memory/feedback_a2a_relay_reliability.md` (extract from main's `ws-history/<sender-agentId>.jsonl` and forward to the affected recipient) remains a *valid operational fallback* even after v0.4 ships, for the case where the server's pending queue exceeds its cap or the recipient stays disconnected past `RelayUnreachable`. The workaround does NOT bypass v0.4 dedup ظ¤ the broker-forward emit carries the original `msgId`, so the recipient's seen-msgId set deduplicates against the original delivery attempt. The workaround is the manual escalation path that runs alongside automatic redelivery.

**Phase 1 ship (v2.5.11)** ظ¤ this v0.4 RRP draft codifies the contract. Reference-implementation work (server.cjs pending queue + redelivery scheduler + bridge dedup LRU) follows in v2.5.12+ (sequenced after Hermes dataUrl reemit catch + Mode B baseline absorption per user-confirmed time scheduling). The contract above is the spec adopters build against; the reference implementation is the v0.4 dogfood vehicle.

**Calibration signal ظ¤ 2026-06-01 dogfood**: 3 drops / 1 cycle was the threshold that triggered this RRP. Future calibration (post-v2.5.12 reference implementation): the queue cap (256), redelivery threshold (5 min / 30s), max attempts (3), and dedup-LRU (1024 / 1 hour) are *provisional defaults*; expect tuning after 1-2 weeks of v0.4 telemetry. Adopters report observed redelivery counts + dedup-hit rates + `RelayUnreachable` emit rates as the empirical anchors.

**(g) Bridge inbox-append commitment (v0.5 addition)** ظ¤ the v0.4 design conflated *pending-queue clear* with *agent application-tier acknowledgement*: the server kept a pending entry until the recipient's agent layer emitted an `AckProcessed` (the "body has been surfaced into working context" assertion). Operationally that introduced a class of false-positive `RelayUnreachable{commitment-ack-absent}` errors ظ¤ when a recipient agent was *idle* between turns, or *busy processing an unrelated task*, the server's redelivery threshold elapsed without an `AckProcessed` and the sender saw an "unreachable" signal for a message that had in fact been *durably written to the recipient's inbox file*. The 2026-06-02 dogfood (relay RRP ┬د13.13.2 v0.4 hub adoption immediately after ship) surfaced this: a normal `Ack` from EG ظْ main was redelivered 3├ù ظْ `RelayUnreachable{commitment-ack-absent}` despite the body having reached main's inbox on the first attempt, because main's agent layer did not happen to emit a commitment-tier `AckProcessed` for it.

The v0.5 resolution: the **recipient's bridge** (the WS bridge process that owns the inbox file ظ¤ distinct from the agent layer that *reads* the inbox) MUST emit `AckProcessed { ackFor: msgId, tier: "delivered-persist" }` *immediately upon appending the inbound CUSTOM envelope to its inbox file*. The bridge does not wait for the agent layer to read or process the body ظ¤ the moment the body is durably persisted to the inbox (the per-┬د13.16.6 cursor-tail watching anchor), the bridge has fulfilled its half of the at-least-once contract and emits the commitment signal on its own authority. The server treats this `tier: "delivered-persist"` ack as the canonical *pending-queue clear* trigger, exactly equivalent in semantic weight to v0.4's commitment-tier ack but emitted by the transport layer that actually owns the persistence guarantee, not by the application layer that owns the *interpretation* of the body.

This removes the false-positive class entirely: every successfully-delivered envelope produces an immediate `delivered-persist` ack within milliseconds of the bridge's inbox write, so the server's pending entry clears before the redelivery threshold elapses for the normal-case path. Redelivery only fires when the bridge is not running, has crashed, or the recipient is not currently `AgentList`-present ظ¤ the actual reachability failure modes the v0.4 RRP was meant to address.

**(h) Tier semantics ظ¤ v0.5 vocabulary (renames v0.4's implicit conflation)** ظ¤ v0.5 distinguishes three named tiers on the response side of an `AckProcessed`:

| Tier name | Emitter | Trigger | Pending-clear |
|---|---|---|---|
| `delivered` | server (proxy) | wire frame received from sender; relayed to recipient | no (transport only) |
| `delivered-persist` | recipient **bridge** | inbox-file append complete; durable | **yes** (canonical v0.5 trigger) |
| `commitment` | recipient **agent** | body surfaced into agent's working context; agent will act | redundant after `delivered-persist` (no-op at server) |

Application-tier outcomes (`Report` / `DONE` / `BLOCKED` per ┬د13.13) are preserved verbatim; they live on the *outcome* dimension, separate from the pending-clear dimension. An agent that emits `AckProcessed{tier:"commitment"}` (the v0.4 name) is still spec-compliant ظ¤ the server simply observes that the pending entry is already cleared (by `delivered-persist`) and treats the commitment ack as informational. v0.4 receivers that emit `AckProcessed` without an explicit `tier` field are interpreted as `commitment` for back-compat, and the server falls back to pending-clear on first such ack (the v0.4 behaviour) ظ¤ which means a v0.4 receiver paired with a v0.5 server continues to operate at v0.4 semantics, including the false-positive class. The escape from false positives requires the v0.5 *bridge* (the receiver-side process), not a v0.5 server.

**(i) Migration path v0.4 ظْ v0.5** ظ¤ the wire envelope shape is unchanged (`msgId` remains the only required v0.4 field; v0.5 adds the optional `tier` field to `AckProcessed.value`). Per-component update:

- **Server** ظ¤ gains the rule "any `AckProcessed{tier:"delivered-persist"}` clears the pending entry, in addition to the v0.4 commitment-tier clear". Optional: add per-target redelivery-threshold override allowing operators to disable redelivery (`RELAY_REDELIVERY=off`) while v0.5 bridges roll out across the deployment.
- **Bridge** ظ¤ adds the inbox-append hook: on every successful inbound CUSTOM body append, emit `AckProcessed{ackFor:msgId, tier:"delivered-persist"}` to the sender. Composes with the v0.4 ┬د13.13.2(c) seen-msgId LRU dedup: a duplicate msgId still emits `AckProcessed{ackFor:msgId, dedupHit:true}` and discards the body without an inbox-append (no `delivered-persist` for duplicates ظ¤ the original delivery already produced one).
- **Agent** ظ¤ no change required for v0.5 mechanics; application-tier acks (`Report` / `DONE` / `BLOCKED`) and the optional commitment-tier `AckProcessed` continue to function. The `delivered-persist` mechanism is bridge-layer and invisible to agent-layer code.

The 2026-06-02 dogfood validated v0.5 with main-EG bidirectional: main's bridge (`ws-local-bridge.cjs`) added the inbox-append-commitment hook + restart, and the next EG ظْ main dogfood envelope cleared pending in <1s without redelivery or `RelayUnreachable`. Reference-implementation work for the EG-side bridge (`constellation/reference/runtime/local-bridge.cjs`) ships separately as the v2.5.18 EG-side complement to this spec (the v2.5.16 / v2.5.17 slots were taken by the promo-site cut and the spec patch itself respectively).

### 13.15 Reference-build syntax gate (optional, downstream-adopted)

A complementary discipline to ┬د13.14 ظ¤ that one guards *content* (env-specific token leaks), this one guards *loadability* (a master `.cjs` that fails `node --check` is a non-loading reference that breaks every fresh downstream bootstrap until someone notices). Real driver: a v2.2.x reference shipped with a JSDoc block that closed early on `RUN_*/TEXT_*/TOOL_*` because `*/` inside the comment terminated the block (the `*` adjacent to `/` is an RFC-valid JSDoc trap); downstream adopters hit a `SyntaxError` on Constellation bootstrap. Caught only after a downstream submitted a bug bundle (`_proposals/002`).

**The gate**: `node --check constellation/reference/runtime/*.cjs` must return zero before commit. The implementation is one of two options ظ¤ adopt either, both, or none per project policy. **The EG repo itself adopts option A**; downstream projects may follow either pattern:

- **Option A ظ¤ pre-commit hook** (EG repo's choice). A single `node --check` loop over `reference/runtime/*.cjs` in `.git/hooks/pre-commit`; fails the commit if any file fails. Cheap (~5s wall-clock per commit, only `*.cjs` count) and catches drift *before* it reaches origin. Requires the hook to be installed locally on every clone ظ¤ the seed's `scripts/sync-readme-counts.mjs --install` (or any equivalent) is the natural carrier; EG ships this as `scripts/check-reference-syntax.mjs`. Real-world coverage: only protects developers who installed the hook; CI is the back-stop for the rest.
- **Option B ظ¤ CI workflow** (downstream-adopted). A GitHub Actions / GitLab CI / equivalent job that runs `node --check` on the reference runtime on every push and PR. Doesn't block local commits but blocks merges to the main branch. Higher infrastructure cost (CI minutes), but always-on coverage regardless of which contributor pushed. Recommended for projects where the reference runtime is a shipped public artifact (e.g. a published package or vendored copy in dependent repos).

**Compose with both**: A + B is the safest combination ظ¤ A catches drift in local commits and B catches drift from non-hooked contributors / CI-only environments. The added cost is small (the same `node --check` invocation, twice). For solo projects without CI infrastructure, A alone is the practical baseline.

**Beyond `.cjs`**: the same gate principle applies to any reference-build artifact whose load-failure breaks downstream bootstrap ظ¤ `.py` (`python -m py_compile`), `.rs` (`cargo check`), brewed `.eux` outputs (`node bin/estreux.mjs drift`), etc. The point is *fail at commit time, not at adopter time*.

### 13.16 Runtime operational runbook ظ¤ launch idempotency, orphan handling, channel cleanup

The complement to ┬د13.15 (build-time gate) ظ¤ this is the *runtime-time* discipline for keeping a Constellation deployment stable across restarts, account switches, IDE crashes, and developer machine reboots. The driver was a real downstream incident (per `_proposals/003`): a long live-board session with two external interruptions accumulated 12 same-`agentId` bridge processes; absent any single-instance protection they kicked each other off in an infinite `register(HELLO)` ظْ close ظْ reconnect loop, flooding the board + inbox with ~2,500 `ServerNotice` events until manual OS-level kill.

The bridge's PID lockfile guard (per `_proposals/003` Fix 1, `local-bridge.cjs:L53`) is the in-process line of defense. This runbook covers the **out-of-process layers**: how launcher scripts, restart-orphan cleanup, and channel cleanup compose with it.

#### 13.16.1 Launch idempotency

Any wrapper that starts a Constellation runtime process (`local-bridge.cjs` ┬╖ `server.cjs` ┬╖ `watchdog.cjs` ┬╖ `self-wake-watcher.sh`) MUST be safe to re-invoke. The bridge's own lockfile already handles the worst case (an existing same-`agentId` bridge is killed before the new one acquires the lock), but a launcher that *also* checks the lockfile and the listening port preemptively produces less alarming logs and avoids the brief flap window during the bridge's startup race.

Reference launcher behavior (per `agentId`):

1. Read `constellation/reference/runtime/.<agentId>-bridge.lock` (sanitized filename ظ¤ see `local-bridge.cjs:L53`).
2. If the PID is alive (`process.kill(pid, 0)` succeeds), the launcher's policy decides:
   - **default**: kill the prior PID, log `replacing prior bridge PID N`, start new bridge.
   - **strict**: refuse to start, log the existing PID ظ¤ operator decides.
3. For `server.cjs`: gate on the listening port (e.g. 27878) ظ¤ only one server per port; if `EADDRINUSE` would otherwise be raised, surface it explicitly before launch.

Launcher scripts are project-specific (PowerShell ┬╖ bash ┬╖ npm script ┬╖ systemd unit) ظ¤ they ship as runbook examples, not as canonical reference. The lockfile path + behavior contract is the invariant.

#### 13.16.2 Restart-orphan handling ظ¤ `stop-all` / `status` helpers

Background runtime processes survive IDE / harness restarts at the OS level but the harness loses task tracking, so the harness's normal task API can't reach them. Manual cleanup helpers are required:

- **`status`** ظ¤ list every alive Constellation process: PID ┬╖ agentId (or role) ┬╖ lockfile path ┬╖ bound port ┬╖ parent process ┬╖ start time. Sources: scan `constellation/reference/runtime/.*-bridge.lock` files + check the listening port; cross-reference `node` processes by exact script path (`local-bridge.cjs` ┬╖ `server.cjs` ┬╖ `watchdog.cjs`).
- **`stop-all`** ظ¤ kill every Constellation process and remove its lockfile. **Filter precisely**: `Name=node.exe` (or `node`) AND command-line contains the absolute path of one of the four reference scripts. **NEVER** use a broad `*constellation*` command-line glob ظ¤ it catches IDE shell wrappers and unrelated processes (real-incident anti-pattern: a Windows `Stop-Process | Where {$_.CommandLine -like '*constellation*'}` accidentally killed the parent terminal because the IDE's Constellation channel name was in its title).

The pair makes recovery deterministic: `status` to see what's alive, `stop-all` to start clean, then launchers re-start whatever the deployment needs.

#### 13.16.3 Channel noise cleanup ظ¤ `CloseChannel`

A flap incident leaves the affected channel's history file (`ws-history/<agentId>.jsonl`) and the agent's inbox saturated with `ServerNotice{kind:'online'}` + `Status` events. **Browser refresh does NOT clear them** ظ¤ the noise is recorded server-side, not cached client-side. New operators reaching for refresh as a first instinct will be confused.

The cleanup primitive is `CUSTOM/CloseChannel{value:{agentId}}` (board ظْ server):

1. Server deletes `ws-history/<agentId>.jsonl` and removes the in-memory channel state.
2. Server broadcasts `ChannelClosed` so other boards drop the tab.
3. Next time the same `agentId` registers, the channel re-starts clean.

For inbox cleanup, the operator truncates the affected `<scope>/inbox.jsonl` (gitignored, never tracked) ظ¤ the bridge's `cursor` is byte-offset based, so a truncate-to-zero resets the agent's pending queue without losing in-flight WS events. (Truncate must happen *while the bridge is stopped* ظ¤ `stop-all` first, then truncate, then re-launch.)

#### 13.16.4 Server-side HELLO churn dampening (adopted at upstream production ┬╖ reference impl shipped d385561)

The bridge lockfile is the line of first defense, but the server (`server.cjs`) adds a second: detect duplicate `agentId` HELLOs arriving at a rate that suggests a flap and decide *who* to keep.

Adopted behavior (per `_proposals/003` Fix 2 ظ¤ shipped at the upstream live-board production layer in commit d385561, 2026-06-01; reference cp of `server.cjs` into `constellation/reference/runtime/` remains a separate follow-up cycle):

- Per-`agentId` ring buffer of the last `N` HELLO timestamps (default `N=5`).
- If the most recent `K` HELLOs (default `K=3`) span less than `M` ms (default `M=10000`) ظْ `flap` WARN + a `cooldown` window.
- During cooldown, a new HELLO from the same `agentId` is rejected with `ServerNotice{kind:'flap-rejected', incumbentPid: <prev>}`; the incumbent connection survives.
- Cooldown duration: default 30000ms (30s) ظ¤ long enough to absorb a flap burst, short enough that a legitimate restart waits ~30s before its HELLO is honored.
- Env knobs (per d385561): `CONSTELLATION_FLAP_RING_N` (default 5), `CONSTELLATION_FLAP_RING_K` (default 3), `CONSTELLATION_FLAP_SPAN_MS` (default 10000), `CONSTELLATION_FLAP_COOLDOWN_MS` (default 30000). Operator-tunable per deployment if a specific bridge restart pattern needs a different sensitivity profile (e.g., a slow-restart bridge needing N=10 / span=20000).

Why both layers ship: the bridge-side lockfile (┬د13.16.1) covers the common case where the bridge can be process-locked, but it requires fs write access from the bridge process. Foreign-runtime bridges (a non-Node adapter, a sandboxed worker, no-fs sandboxed CI runner, etc.) cannot always lock ظ¤ the server-side dampener is the universal back-stop. They compose: a locked bridge avoids the flap before it starts; the server-side dampener catches it when the bridge can't lock.

(EG-side follow-up surfaced by seq 165: bridge HELLO must carry `pid: process.pid` so the `incumbentPid` field in the `ServerNotice{kind:'flap-rejected'}` carries a real value rather than `null` ظ¤ being implemented in a parallel workstream against `constellation/local-bridge.eux` + the reference runtime cp.)

#### 13.16.5 Composition with ┬د13.13 / ┬د13.14 / ┬د13.15

- **┬د13.13** ack layer is silent by design ظ¤ it does NOT generate the `ServerNotice` flood that ┬د13.16 cleans up. The flood is a *transport-layer* artifact (HELLO churn from same-`agentId` register), so ┬د13.16 fixes are also transport-layer (lockfile + launcher + orphan kill); they're orthogonal to the application-layer ack mechanism.
- **┬د13.14** (env-specific token redaction) applies to runbook examples too: `stop-all` / `status` stubs MUST NOT inline real agentIds, port numbers, or workspace paths from the upstream's environment; use `${AGENT_ID}` / `${PORT}` / `${WORKSPACE}` substitution. The lockfile filename's `agentId` sanitization (`replace(/[^\w.-]/g, '_')` per `local-bridge.cjs:L53`) is the in-code form of the same discipline.
- **┬د13.15** (build-time syntax gate) catches a non-loading master that would otherwise reach adopters; ┬د13.16 (runtime ops) catches the *operational* equivalent ظ¤ a loadable but mis-launched master that produces a flap storm at adopter sites. Build gates ship the right bytes; runtime gates ship the right *process topology*.
- **┬د13.16.6** (watcher liveness probe, below) extends ┬د13.16.1-.4: the launch/orphan/cleanup fixes ensure a watcher *starts cleanly*, but they don't guarantee it *stays armed*. The probe discipline closes the gap.

#### 13.16.6 Watcher liveness probe discipline

When the adopter agent parks in a "wait for external response" cycle ظ¤ board `UserPrompt` waiting on a remote agent's `Delegate` reply, an upstream worker's job-done event, an inbox-polling rearm chain ظ¤ **a watcher/background task that polls the inbox.log silently dies more often than expected**. Symptoms: a `Ping`/`AckProcessed`/`Delegate` arrives at the inbox, but the adopter never surfaces it because the watcher's rearm chain broke (process crash, user-interrupt, exceeded rearm ceiling, harness garbage-collection of the background task).

A "watch maintained" claim *without an explicit liveness check* is the silent-stall class of bug ظ¤ work looks healthy because no error fires, but hours pass with the response unread. The user is typically the first to notice (`"هإّنï╡ هآ¤هû┤?"` / `"هآ£ هـê هؤهدهùش?"`) ظ¤ that is the failure mode this discipline exists to prevent.

**Probe rules** (apply whenever the adopter is parked waiting on an external response):

- **Inbox mtime probe** ظ¤ re-tail `assets/collab/inbox.log` and check its mtime vs current wall-clock. Constellation watchers typically rearm every 40-45 min; an inbox.log untouched for >60 min while the adopter is in a wait cycle is suspect (either no inbound *or* watcher dead ظ¤ disambiguate via the next rule).
- **Rearm log probe** ظ¤ the watcher's task output should contain a recent `watcher re-armed @ <Z>` marker (one per rearm cycle). If the last marker is >2├ù the rearm interval behind wall-clock, the watcher is parked or crashed.
- **Cross-check inbound side** ظ¤ when expecting a reply, peek at the watcher's task output (`WAKE` / `REARM` markers). Last marker hours behind wall-clock = dead watcher, not "no inbound".
- **Re-arm explicitly when stale** ظ¤ do not "hope" a dead watcher recovers. Spawn a fresh watcher cycle on the same polling target, note the new task ID, and mark the old one dead in the agent's working state.
- **Surface staleness immediately** ظ¤ tell the user the watcher went stale, how long the gap was, and the action taken (re-arm or escalate). Never continue to claim "watch maintained" silently after detecting staleness; that converts a recoverable miss into an unrecoverable one.

**Discipline cadence**: probe at least once per **independent work cycle** taken during the wait, or at least every **~30 min wall-clock** if the agent is otherwise idle ظ¤ whichever fires first. Do NOT conflate todo state (`[in_progress] watcher`) with task liveness (the bash background task actually running). Todo state is the agent's *claim*; task liveness is the *truth*. Verify the truth before reusing the claim.

**Why this belongs in the runbook, not just agent habit**: an adopter without an explicit probe discipline will treat "watcher launched once" as sufficient and lose responses for hours. The probe is part of the *operational contract* for Constellation watchers ظ¤ same tier as `stop-all` / `status` (┬د13.16.2) and the lockfile (┬د13.16.1).

**Turn-end mandatory rearm ظ¤ the discipline that prevents the multi-cycle failure mode**: probe-without-rearm catches a dead watcher but does not prevent the NEXT cycle from being dead. At the close of EVERY user-facing response (just before emitting the final text), the agent MUST run a 1-shot Bash that:

1. Reads `inbox.log` mtime and the most recent watcher task's output mtime.
2. **Unconditionally spawn a fresh background watcher** (`run_in_background: true`) with the same polling target ظ¤ every turn-end, regardless of whether an existing watcher appears alive.

**Unconditional, not conditional ظ¤ the cost-asymmetry argument**: the agent's active-liveness check on the prior watcher is unreliable. It depends on the agent (a) correctly reading the bg task's output mtime, (b) correctly interpreting the task state (alive vs paused-by-SIGSTOP vs cleanly-exited vs crashed), and (c) not mistaking a harness-paused bg task for a dead one (the wall-clock-vs-task-clock mismatch documented below makes "mtime within rearm interval" itself an unreliable liveness signal). Cost of a false-positive spawn (the agent rearms when a live watcher already exists) = one extra task slot + a few log lines, both harmless. Cost of a false-negative miss (the agent skips rearm because it judged the prior watcher live when in fact it was paused/dead) = the silent-stall class ظ¤ hours of lost inbound until the user surfaces the stall. The asymmetric cost structure (harmless false-positive vs hours-of-lost-work false-negative) makes unconditional spawn the correct rule: pay the trivial double-arm cost every turn rather than gamble on a liveness check that is structurally unreliable. Same-base redundancy at the same `base` cursor does NOT add inbound coverage for the windows the prior watcher was already covering (the a7911fd armed-window-blind-spot diagnosis still holds ظ¤ redundant watchers at the same `base` share the same blind spot), but it ALSO does not subtract: the fresh spawn IS a fresh `base` after the prior watchers complete, so future inbound DOES get covered by the new watcher's cursor. There is no scenario where unconditional spawn loses coverage relative to conditional.

*(Honest-history note: commit 0185b8d introduced a conditional-rearm rule on the rationale that "one live watcher is sufficient" once the cursor-tail fix was in place. That decision is **RETIRED as of this turn (Delegate seq 109, msgId `m-mpthobk3-108`, 2026-05-31)**. The cursor-tail fix remains the structural coverage layer for armed-window gaps ظ¤ it is not retired ظ¤ but it does not justify trusting the agent's liveness check on the prior watcher, which is the part of 0185b8d's reasoning that fails in practice. Conditional rearm requires the agent to correctly judge "is the prior watcher alive?" at every turn-end; the failure mode this section exists to prevent is precisely the absorbed multi-cycle state where the agent's volitional judgment is unreliable. Unconditional spawn removes the judgment from the path.)*

**Fail mode this addresses** (observed 2026-05-31): agent processed inbound seq 70 ظْ entered a multi-cycle work loop (workflow ├ù 3 + commit ├ù 2 + a follow-up patch) ظْ did not explicitly rearm at any cycle boundary ظْ silently lost seq 73 (catalog review approval) and seq 77 (new user task: live-board UI + DB delegation) for ~40 minutes until the failure surfaced externally. The previous `┬د13.16.6` probe rule was correct but insufficient: it required the agent to *think to probe* at the right moment, which an agent absorbed in multi-cycle synthesis does not reliably do. Making rearm a **turn-end ritual**, not a *probe-then-decide* judgment call, removes the failure path.

**Cadence interaction with the probe rule**: the probe rule (above) catches a dead watcher when the agent is already awake. The turn-end ritual prevents the watcher from dying between turns in the first place and prevents board/queue drift while the agent is awake. Both are required ظ¤ probe is the PRIMARY catch mechanism for the armed-window blind spot (the gap between the prior watcher's `base` cursor and the next watcher's spawn); the unconditional rearm is the forward-looking catch mechanism for the next idle window. Neither is a "safety net" for the other ظ¤ they cover *disjoint* windows: the probe catches inbound that already landed below the new watcher's `base`; the rearm catches inbound that will land after the new watcher arms. As of policy #442 (Delegate seq 109, 2026-05-31) the turn-end ritual is a **5-element mandatory checklist** every user-facing response MUST run before emitting final text: (1) inbox.log mtime / cursor-tail probe, (2) unconditional watcher rearm, (3) board-state real-time update (`done` / `current` / `updatedAt`), (4) BOARD_STALE gate handoff (┬د13.16.7 ظ¤ main only; the gate itself runs on the watcher's rearm path, but the agent acknowledges any `WAKE(BOARD_STALE)` from the prior cycle here), (5) planned-queue scan with "start a progressable item now, don't idle-wait while work remains" default. As of policy seq 120 (msgId `m-mptjjxmo-119`, 2026-05-31) the checklist is **extended to 6 elements** with the addition of **(6) pre-large-work context materialization** (see paragraph below). None of the six is optional; none is gated on a volitional judgment the agent makes about whether the element is needed this turn.

**Mandatory ordering ظ¤ element 1 (cursor-tail probe) STRICTLY BEFORE element 2 (unconditional rearm)**: the probe is NOT diagnostic telemetry; it is NOT "after-the-fact" verification; it is the **gating step** that fills the armed-window blind spot the new watcher structurally cannot cover. A fresh watcher armed at `base = current line count` is permanently blind to any inbound below `base` ظ¤ including inbound that landed while the prior watcher was suspended (SIGSTOP'd by harness during agent-active turn, per the wall-clock-vs-task-clock mismatch below). The rearm alone DOES NOT recover that inbound; only the cursor-tail does. Concrete step list every turn-end MUST execute, in this order:

1. **(a)** Tail `inbox.log` from the agent's last-surfaced cursor (NOT from the prior watcher's `base`) ظ¤ this is the agent's own cursor maintained across turns, not the watcher's.
2. **(b)** Surface any meaningful inbound found in the tail ظ¤ `Delegate`, `UserPrompt`, `WorkerReport`, `WorkerAck`, `AgentHello`, `Handoff` ظ¤ into the current response context. If meaningful inbound is found, the turn does NOT end on the current plan; it ends on a response to the newly-surfaced inbound.
3. **(c)** Advance the agent's cursor to the last line read.
4. **(d)** THEN run the unconditional rearm (element 2).

Skipping (a)-(c) and going straight to (d) loses any inbound that landed during the prior watcher's suspend window and is **structurally invisible to the new watcher** ظ¤ the new watcher's `base` is past that inbound's line number, so the polling loop will never see it. Re-arm without probe is forward coverage without backward catch; it is not equivalent to running both.

**Wrong order ظ¤ anti-pattern (observed 2026-05-31)**: agent ran element 2 (unconditional rearm) first WITHOUT first running element 1 (cursor-tail probe). seq 123 (Status) and seq 127 (Report: DB P1 ship + ┬د13.19 P2 spec-delivery query) arrived while the prior watcher was active/suspended; the new watcher armed at `base = current line count` (above both inbound's line numbers), putting both inbound below its base ظ¤ permanently invisible to the new watcher. The agent only ran the cursor-tail probe manually after the missed inbound was surfaced, recovering the 2 missed messages. The failure was NOT the rearm (it worked correctly for forward coverage); the failure was the ordering ظ¤ running the rearm without first running the probe to catch what was already below `base`. The fix is the ordering reinforcement in this section, not a new mechanism: element 1 was already in the 6-element checklist; what was missing is the MANDATORY ORDERING that element 1 runs before element 2, every turn, without exception.

**Filter discipline ظ¤ exclude self-emission AND server-replay noise (ev-agnostic, name-only)**: the watcher's "meaningful inbound" filter is a **single-stage name-only** pipeline ظ¤ a noise blocklist followed by a meaningful allowlist, with NO `ev:"inbound"` scope gate ahead of them. The blocklist and allowlist memberships are defined in **┬د13.16.9 (Message classification by intent)** ظ¤ the noise blocklist === board-directed allowlist + transport-tier; the meaningful allowlist === A2A-intent allowlist. ┬د13.16.9 is the SSoT for the name enumeration; this section references it by anchor. **Board state real-time update ظ¤ the third turn-end ritual element**: alongside the inbox.log mtime probe and the unconditional watcher rearm, the agent MUST verify that the live-board `state.json` reflects the just-completed work *before* emitting the final response text. 3. **`updatedAt` bumped to wall-clock** ظ¤ the top-level `state.json.updatedAt` field is bumped to current wall-clock (ISO-Z), not left at the last write's timestamp. A stale `updatedAt` makes the dashboard's "last activity" indicator lie about the agent's real liveness.

**Planned queue scan ظ¤ the fifth turn-end ritual element**: alongside the four inbound/outbound-liveness elements above (inbox.log probe ┬╖ unconditional watcher rearm ┬╖ board state real-time update ┬╖ BOARD_STALE gate per ┬د13.16.7), the agent MUST scan its **planned/pending work queue** at every turn-end *before* deciding whether to enter idle wait. #### 13.16.7 BOARD_STALE gate ظ¤ pre-rearm board freshness check

**The gate** (mechanization of ┬د13.16.6's third turn-end ritual element ظ¤ board state real-time update): at the *start* of every watcher re-arm cycle, before entering the polling loop, the watcher compares the git HEAD commit time against the last commit time that touched `state.json`. Concretely: `git log -1 HEAD --format=%ct` (the working tree's tip commit epoch) versus `git log -1 -- state.json --format=%ct` (the most recent commit that included a `state.json` change). The gate **fires** when HEAD's commit time is *strictly greater than* the state.json commit time ظ¤ i.e. there exists at least one commit after the last board-refresh commit that did not refresh the board. The gate **passes** when the two are equal ظ¤ i.e. the most recent HEAD commit itself included the `state.json` update, so the board is current as of HEAD. When the gate fires, the watcher does NOT re-arm; it exits immediately with a `WAKE(BOARD_STALE)` event surfaced into the watcher's task output (and into `inbox.log` as a self-emitted wake marker, per the ┬د13.16.6 self-emission filter discipline). The next turn-start will read the wake, the agent will write the missing `state.json` deltas and commit them, and only then will a fresh watcher arm. **No time buffer, no env-configurable threshold** ظ¤ the comparison is identity-based on commit ancestry, not offset-based on wall-clock.

**Why mechanize, not just discipline**: ┬د13.16.6 already specifies the board-state update as the third turn-end ritual element, and that rule is correct. But discipline based on agent volition was observed to fail **twice** (2026-05-31) when the agent was absorbed in multi-cycle work (workflow chain + commit pipeline + follow-up patches) ظ¤ the very absorption that makes the rule necessary is the same absorption that makes volitional compliance unreliable. Volitional discipline does not survive the absorbed state. Moving the check off the agent's turn-end ritual and onto the watcher's re-arm path converts the failure mode from "agent forgets to update board ظْ board drifts indefinitely until human notices" into "watcher refuses to re-arm ظْ agent is forced to update board on the next wake, at the latest within one rearm-interval cycle". The gate cannot be forgotten because the agent is not the one running it.

**Scope ظ¤ main only**: the gate runs only when the watcher's `agentId === WS_PRIMARY_AGENT` (i.e. the main agent that owns the live-board `state.json` write). Workers do NOT own `state.json` ظ¤ their outbound is `WorkerReport` to the main inbox, and the board reflects worker progress via main's relay write ظ¤ so a worker watcher's HEAD-vs-state-commit-time comparison would be meaningless (workers commit to their own worktrees on their own cadence, unrelated to the board's `state.json` history). Worker watchers skip the BOARD_STALE check and re-arm normally.

**Self-trap-avoidance rationale ظ¤ why no time buffer**: an earlier draft of this section used a 120 s time-buffer rule (`headEpoch ظêْ stateUpdatedAtEpoch > 120 s` fires the gate). That draft **self-trapped**: the board-refresh commit itself takes wall-clock time to author (commit hook + push + any in-flight workflow), which moves HEAD's commit epoch past the `state.json.updatedAt` value the agent just wrote ظ¤ by a margin equal to the workflow's wall-clock cost. Any workflow longer than the buffer (and many normal multi-cycle workflows easily exceed 120 s) would therefore keep `headEpoch ظêْ updatedAt > 120 s` even immediately after a correct, conscientious board refresh, leaving the gate fired forever and the watcher refusing to re-arm despite the board being current. The fix is to switch from an *offset-based* comparison (wall-clock delta against a tolerance) to an *identity-based* comparison (commit ancestry: did the latest HEAD commit include `state.json`?). With the head-vs-state-commit-time form, the board-refresh commit advances both `git log -1 HEAD --format=%ct` and `git log -1 -- state.json --format=%ct` to the *same* value in lockstep ظ¤ the very act of refreshing the board passes the gate, because the refresh commit IS the HEAD commit that touched `state.json`. There is no wall-clock margin to overshoot, so no self-trap.

**Composition with ┬د13.16.6**: the third turn-end ritual element (board state real-time update) remains the *primary* control ظ¤ the agent should still verify the board reflects the just-completed work *before* emitting the final response. The BOARD_STALE gate is the **safety net** that catches the case where the primary control fails (agent absorbed in multi-cycle work, forgot the ritual). Same two-layer pattern as inbox liveness: probe (┬د13.16.6, agent-side) is primary, turn-end mandatory rearm (┬د13.16.6.1 ritual) is the safety net; here, turn-end board update is primary, BOARD_STALE re-arm gate is the safety net. Both layers required for the same reason: the primary control is the right thing to do every turn, the safety net is what saves the cycle when the primary fails.

#### 13.16.8 A2A communication priority discipline ظ¤ outbound messages (and their prerequisites) take priority over other planned actions

**The discipline**: whenever an external state change occurs ظ¤ user steering, a newly-arrived task, an external event (counterpart `Delegate` / `Report` / `Status` / `BlockerNudge` inbound, cycle restart with a queued plan) ظ¤ and the resulting work plan contains at least one outbound A2A message to a counterpart agent (e.g., outbox push to `main`, `Delegate` to a worker, `Report` back to a requester, `BlockerManifest` emission), the agent MUST execute the following ordering: (1) **scan the new plan's action list and identify every action that is itself an outbound A2A message** to a counterpart; (2) **also identify the prerequisite work that the counterpart needs in order to act on the message** ظ¤ any commit-push of referenced docs the counterpart will fetch (a `Delegate` payload that points to a doc URL is unusable until the doc is reachable at that URL), any payload that needs assembling, any `state.json` or live-board update the counterpart's decision depends on; (3) **move both the message itself and its prerequisite work to the head of the action queue**, ahead of any other planned action; (4) only after the message has been emitted do the remaining planned actions proceed.

**Two technical rationales**:

(a) **Drift-containment**. If the EG side updates a spec, a state field, or a doc *before* notifying the counterpart, the counterpart continues operating on a now-stale baseline for the entire duration of the unnotified gap. Any work the counterpart performs in that gap risks landing on a superseded baseline ظ¤ wasted effort, divergent artifacts, follow-up reconciliation cost. Sending the message first ظ¤ even while other local actions remain queued ظ¤ lets the counterpart pause, redirect, or re-fetch at the earliest possible moment. The drift window collapses from "EG's local-work duration" to "round-trip latency", which is typically orders of magnitude smaller and bounded by transport, not by EG's queue depth.

(b) **Counterpart-start-time minimization**. When the new plan obligates the counterpart to start a piece of work ظ¤ e.g., "main runs DB benchmarking starting from Mode B entry", "worker takes over the reference-runtime cp track", "collab reviews the new RRP" ظ¤ every minute EG spends on local actions before emitting the trigger is a minute the counterpart hasn't started. Total wall-clock to cycle completion is `message-emit-time + counterpart-work-duration`; advancing the emit time directly shortens the cycle. Local EG actions that don't gate the message can happen in parallel with the counterpart's work once the message is out, but cannot recover the head-start lost by emitting the message late.

**Symmetry**: the discipline applies to any role classified by ┬د13.16 (`main` / `local` / `collab` / `upstream`) when that role initiates a cross-agent change. A worker reporting back to main, main delegating to a worker, EG reporting back to an upstream collaborator, an upstream agent steering a downstream ظ¤ all four directions follow the same priority rule. The rule is about *the act of initiating a cross-agent state change*, not about who initiates it.

**Composition with existing ┬د13.x ظ¤ no override of orthogonal gates**:

- **┬د13.14 redaction** still gates the message *content* (placeholder substitution for non-public identifiers, technical-only phrasing). The priority rule decides *when* the message goes; ┬د13.14 decides *what's in it*. Both apply; neither bypasses the other.
- **┬د13.17 main-chat structured-choice ban** still gates the message *destination* (board-mediated route for structured Q/A, no inline option UI in main-chat). The priority rule does not authorize routing a structured choice into main-chat just because it's the head-of-queue item; the ┬د13.17 routing decision is made first, then the chosen destination's emission is what gets priority.
- **┬د13.20 BlockerManifest emission**: a blocker that triggers a `BlockerManifest` is itself an outbound A2A communication and gets the same priority as any other A2A message. When a turn-end ritual identifies a blocker that needs `BlockerManifest` emission *and* the plan also contains other actions, the emission goes first.
- **Destructive-operation authorization gates** are not bypassed: if the prerequisite work for a message is itself a destructive operation (e.g., a force-push the counterpart will need to fetch), the destructive-operation authorization path still applies. The priority rule reorders the queue; it does not relax authorization requirements.

**Composition with ┬د13.16.6 turn-end ritual ظ¤ independent schedule**: the cursor-tail probe (element 1) and the unconditional watcher rearm (element 2) run on their own schedule ظ¤ every turn-end, regardless of whether the current plan contains an outbound message. The A2A communication priority discipline governs *new outbound messages triggered by the current cycle's work plan*; it does not modify the watcher-rearm pattern, which is a separate liveness-maintenance loop. Element 5 (planned-queue scan) interacts with this rule in the obvious way ظ¤ when the scan classifies the next progressable item as an outbound A2A message (or as prerequisite work for one), the priority rule confirms that item as the head-of-queue choice.

**Composition with ┬د13.18 recommend+proceed**: this priority rule sits above ┬د13.18. Even a 1-line "recommend+proceed" announcement is itself an outbound message in the main-chat surface; if its emission unblocks a counterpart (e.g., a `Status` update that the counterpart's `ReviewSLAAck` was waiting on, a board-side `Decision` that downstream agents will read), the 1-line emission takes priority over other planned actions per the same drift-containment and counterpart-start-time rationales.

**Failure mode this prevents**: the agent receives a state change, plans 5 actions of which action #3 is "send `Delegate` to worker", and executes 1 ظْ 2 ظْ 3 ظْ 4 ظْ 5 in plan-order. The worker's start time is delayed by the wall-clock cost of actions 1 and 2, and if action 2 touched a spec the worker reads, the worker would have operated on stale state for the entire interval. The priority rule reorders to: prerequisite-of-3 ظْ 3 ظْ (1, 2, 4, 5 in their natural order). Same final state, counterpart starts at the earliest possible moment, drift window bounded by round-trip latency.

#### 13.16.10 Pre-send inbound check discipline (read-before-write, git push-before-pull analogy)

**The rule**: every outbound A2A message MUST be preceded by a **cursor-tail probe** of the agent's last-surfaced inbox cursor. If new meaningful inbound exists between the cursor and the current `inbox.log` line count, the agent MUST surface that inbound first and either (a) **incorporate** the new information into the outbound payload, or (b) **defer** the outbound entirely if the new inbound supersedes the planned send. Only after the probe is clean (cursor up-to-date, no new meaningful inbound) does the outbound proceed. The probe fires **before each outbound emission**, not once per turn.

**The git push-before-pull analogy ظ¤ and why Constellation A2A is stricter**: the discipline is the same shape as `git pull` before `git push`. A `push` issued without a fresh `pull` races against upstream changes. Constellation A2A is **worse than git** in this regard: git rejects a non-fast-forward push at the transport layer and forces the operator to pull; Constellation A2A has **no in-flight rejection** ظ¤ a stale-context send simply lands on the server, is relayed to the counterpart, and may overwrite or contradict a fresh inbound the sender never saw. The drift is silent. The pre-send probe brings the same read-before-write rule explicitly into the A2A protocol, where the transport itself does not enforce it.

**Concrete failure mode**: counterpart emits `Report X` at `t=0` carrying new context (a state change, a decision, a baseline shift). Agent has `Report Y` queued for `t=1` framed from pre-`t=0` context. If agent sends `Y` without first surfacing `X`, `Y` arrives at the counterpart with stale framing ظ¤ the counterpart now has to reconcile two diverging threads (its own `X` and the sender's `Y` that does not yet incorporate `X`), often requiring a follow-up correction outbound from the original sender. The fix is one-way: agent always reads before writing.

**Why ┬د13.16.6 element 1 (cursor-tail probe at turn-end) is insufficient alone**: ┬د13.16.6's cursor-tail probe runs at the *end* of a turn ظ¤ at the turn boundary. A single turn can contain **multiple outbound A2A sends** interleaved with local work (commits, file edits, agent dispatches, subagent fan-outs, tool results that themselves surface new inbox state). New inbound can arrive *during* a turn between any two outbound sends. The turn-end probe catches what landed across the turn boundary; it does **not** catch what lands mid-turn. ┬د13.16.10 is the **mid-turn probe** ظ¤ fired before *each* outbound A2A emission. The two probes cover disjoint windows: turn-end covers the inter-turn gap (post-emit-last ظْ next-turn-start); pre-send covers the intra-turn gap (turn-start ظْ first-emit, and between any two emits within the turn).

**Operational shape** (the four steps every outbound A2A emission MUST run, in this order):

1. **Probe** ظ¤ before each `cat >> outbox.jsonl` / `appendFileSync(outbox.jsonl, ظخ)` / equivalent outbound emission: read the current `wc -l assets/collab/inbox.log` line count, compare to the agent's last-surfaced cursor, scan the new lines (cursor ظْ current) through the ┬د13.16.6 v2.4.16 ev-agnostic single-stage name filter using the ┬د13.16.9 A2A-intent allowlist (the canonical meaningful-inbound classifier).
2. **Surface (if meaningful inbound found)** ظ¤ bring the new inbound into the current turn's working context before any outbound. Either revise the planned outbound payload to incorporate the new information, or defer the outbound entirely if the new inbound makes the planned send stale, wrong, or obsolete.
3. **Emit (only if probe is clean OR after surfacing + revision)** ظ¤ proceed with the outbound. If the probe was clean, the outbound proceeds as planned; if the probe surfaced meaningful inbound and the agent revised the payload, the revised outbound proceeds; if the agent deferred, no emission occurs this step.
4. **Advance cursor** ظ¤ after the outbound emits, advance the agent's last-surfaced cursor to the line count read at step 1 (the pre-send probe boundary). The next pre-send probe begins from this new cursor.

**Composition with existing ┬د13.x**:

- **┬د13.16.6 turn-end ritual element 1** (cursor-tail probe at turn-end) ظ¤ operates at the *turn boundary*. ┬د13.16.10 operates at *each outbound emission*. Both are mandatory; neither replaces the other. The turn-end probe catches what landed across the turn boundary (between the last emit of turn N and the start of turn N+1); the pre-send probe catches what lands mid-turn (between turn-start and the first emit, and between any two emits within the same turn).
- **┬د13.16.8 A2A communication priority** ظ¤ when an outbound message is priority-bound (the message + its prerequisite work moved to head-of-queue per ┬د13.16.8), the pre-send probe sits **between the prerequisite work and the message emission**: `prereq ظْ probe ظْ emit`. If the probe surfaces new inbound, the priority rule's drift-containment rationale becomes even sharper: a stale-context message defeats the very drift-containment ┬د13.16.8 was meant to provide. The probe is a guard on the very rule ┬د13.16.8 ships.
- **┬د13.16.9 channel classification** ظ¤ the pre-send probe filter applies the ┬د13.16.9 A2A-intent allowlist (the same allowlist the watcher uses in ┬د13.16.6). Board-directed names don't carry context that can race (display-tier UX surface), but the discipline is uniform: **probe before every outbound regardless of intent class** ظ¤ the cost is one `wc -l` + one `grep` invocation, both negligible. Uniformity beats per-class branching at the disciplinary layer.
- **┬د13.21 fresh-context defer = anti-pattern** ظ¤ the pre-send probe is **NOT a deferral**; it is *current-cycle* read-before-write. Deferring an outbound *based on probe results* (the new inbound makes the planned send obsolete, so the planned send is dropped or revised in the same cycle) is a *current-cycle* decision to drop or revise, **not** a ┬د13.21 violation. ┬د13.21 forbids deferring planned work to a hypothetical fresh-context cycle; ┬د13.16.10 authorizes dropping or revising a planned outbound *in the current cycle* when new inbound makes it stale. The two compose cleanly: revise-or-drop now (┬د13.16.10), never defer to a phantom future self (┬د13.21).

**Forbidden anti-patterns**:

- **Skipping the probe because "the inbox was just checked at turn start"** ظ¤ multiple outbounds across a single turn each require a fresh probe. The turn-start probe is one read; the next outbound is an independent decision point with its own race window against new inbound. The agent does NOT get to amortize one probe over multiple emissions.
- **Skipping the probe for "small" or "low-priority" outbounds** ظ¤ the probe cost is negligible (one `wc -l` + one `grep`). The skip-pattern erodes the discipline; if probing is conditional, the agent will judge wrongly under load and the silent-drift failure mode returns. Probe is unconditional.
- **Probing but ignoring the result** ظ¤ if meaningful inbound is found, it MUST be surfaced before any outbound that would otherwise drift past it. Probing and then emitting anyway, on the rationale that the planned send is "ready" or "already drafted", defeats the rule: the cost of the probe was paid but the safety the probe provides was discarded.

**The discipline in one line**: read the inbox from the cursor, surface anything meaningful, then write. Every outbound. No exceptions.

**Cycle-end + user-facing emit-direct probe extension (v2.5.2)**: the ┬د13.16.10 pre-send probe above fires before every *outbound A2A emission*. A separate failure surface ظ¤ observed twice during the 2026-06-01 cycle ظ¤ sits at the *cycle-end user-facing assistant text emit*. If meaningful inbound arrives between the last outbound A2A emit and the cycle-end user-text emit (the window where a turn closes with a response to the user rather than another A2A emit), the agent silently misses it: the pre-send probe rule does not fire (no outbound emit follows), and the user-facing response is already framed without the new context. The next watcher WAKE catches the inbound, but by then the user has already seen a response shaped from stale framing.

**The extension** (two rules added to the ┬د13.16.10 discipline; the existing four-step operational shape above is unchanged for the outbound-A2A surface):

1. **Cycle-end final cursor-tail probe (assistant text emit closes the turn)**: just before the user-facing assistant text emits (the act that closes the turn), the agent MUST run a final cursor-tail probe ظ¤ the same probe shape as the pre-send probe (┬د13.16.6 element 1 form, A2A-intent allowlist filter). If meaningful inbound is found, the agent SHOULD either (a) revise the in-progress user-facing response to incorporate the new inbound (preferred when the new inbound is directly relevant to the response under construction), OR (b) explicitly note in the response that the new inbound was caught and will be handled in the next cycle (acceptable when the new inbound is on a different track from the current response). What the agent MUST NOT do is emit the user-facing response without checking ظ¤ the cycle-end probe is unconditional, same cost asymmetry argument as ┬د13.16.6 element 2 (the probe costs `wc -l` + one grep; the silent-miss costs the user surfacing the missed inbound at the next message).
2. **Post-emit watcher rearm (turn-end mandatory)**: the turn-end MUST include a watcher rearm (this restates ┬د13.16.6 element 2 in the ┬د13.16.10 cycle-end context, to anchor the composition). If the prior watcher is alive and still polling, the rearm is harmless (one task slot + a few log lines per ┬د13.16.6 cost-asymmetry argument). If the prior watcher already completed within the current turn (the common case when a long turn's work outlives the prior watcher's polling budget), the rearm spawns a fresh background watcher targeting the current inbox cursor. The ┬د13.16.6 element 2 unconditional-rearm rule applies ظ¤ agent volition does NOT decide whether to rearm; the rearm fires every turn-end.

**The pattern name**: "cycle-end probe + watcher rearm" ظ¤ the companion to "pre-send probe-then-inspect-or-abort" from the ┬د13.16.10 outbound-A2A surface. The two patterns cover disjoint windows: pre-send covers the *intra-turn outbound-A2A emit point*; cycle-end covers the *turn-closure user-facing emit point*. Both are mandatory; neither replaces the other; together with ┬د13.16.6 element 1 (turn-end cursor-tail probe at the *prior* turn boundary) they form the three-window coverage that closes all observed silent-miss failure modes ظ¤ pre-turn / mid-turn / post-emit-pre-close.

**Failure-mode incident anchor**: 2 missed-inbound incidents observed in the 2026-06-01 cycle, both downstream of cycle-end probe omission. In both incidents the corrective surface was a main-side reminder at the next user message (the missed inbound was a main-upstream protocol update that arrived during the EG-side cycle's user-facing response composition, and was not caught until the next user message surfaced the gap). The 2-incident anchor is the technical rationale for the cycle-end extension ظ¤ the failure-mode is not hypothetical, it surfaced twice within a single working day on the same protocol axis. The corrective is structural (a new mandatory probe + rearm step at turn-close), not behavioral.

**Composition with ┬د13.16.10 outbound-A2A surface above**: the cycle-end probe is the SAME probe (cursor-tail + A2A-intent allowlist) fired at a DIFFERENT trigger (turn-close vs. outbound-A2A emit). The four-step operational shape (probe ظْ surface ظْ emit ظْ advance cursor) applies symmetrically ظ¤ at cycle-end the "emit" step is the user-facing assistant text emit rather than an outbox push, but the surface-or-defer decision and the cursor-advance step are identical. The discipline-in-one-line above expands: "read the inbox from the cursor, surface anything meaningful, then write ظ¤ every outbound, **and every cycle-end user-facing emit**. No exceptions."

#### 13.16.11 Key management unification ظ¤ `kind`-enum SSoT (v2.5.1)

**Purpose**: this sub-section codifies the unified key-management contract that emerged from the 2026-05~06 dashboard-render dogfood cycle, where three previously separate key concepts (collab key, upstream key, local-key meta) accumulated ad-hoc shape variance across server / bridge / dashboard layers. v2.5.1 names a single `kind` enum that all three layers carry on every key-bearing envelope, so the server's `keyStore`, the `KeyList` broadcast, the `HELLO` handshake meta, and the dashboard's key-management modal all classify keys against the same vocabulary. This is a contract clarification, not a wire-format change ظ¤ the existing `RegisterCollabKey` / `RegisterUpstreamKey` / `RevokeCollabKey` operations (┬د5) keep their shapes; what is added is a top-level `kind` discriminator that travels with the key wherever the key is referenced.

**The `kind` enum (closed set, fail-safe reject on unknown)**:

- **`collab`** ظ¤ external collaborator key (`ck-ظخ` per ┬د5). Issued by main via `RegisterCollabKey{label}`; consumed by the joining agent via the `/join/collab?key=<ck-ظخ>` URL ظْ `collab` role ┬╖ `group:collab`.
- **`upstream`** ظ¤ upstream agent key (`uk-ظخ`). Issued by main; consumed via `ws://ظخ/ws?upstreamKey=<uk-ظخ>` ظْ `upstream` role. Per ┬د2 the upstream key authenticates a peer agent (API / gateway), distinct from a local IDE worker that joins without a key.
- **`local-key`** ظ¤ local key for a non-default-environment local agent (e.g., a worker spawned by a sibling main, or a local IDE agent joining a remote server). Distinct from collab (no `group:collab` role assignment) and from upstream (`local` role, not `upstream` role). Optional; many local-IDE adopters never issue local-keys because workers in the same environment join the no-auth `ws://<host>:<port>/ws` path.

Unknown `kind` values MUST be rejected at the server's key-registration boundary (`RegisterKey{kind, label}` returning an error envelope) and at the receiver's `KeyList` consumer (a key with an unrecognized `kind` is logged and skipped, never coerced to a default). The fail-safe-reject discipline mirrors ┬د6.5's `sourceContentRef.kind` enum handling in the A2A PR System RRP ظ¤ closed enums with explicit reject on unknown rather than fallthrough to a default, which prevents silent misclassification (e.g., an unrecognized kind being routed as `collab` would assign `group:collab` to an upstream peer, leaking collab group membership to non-collab agents).

**SSoT surfaces** (the four layers that all read the same `kind` enum):

- **Server `keyStore`** ظ¤ the in-memory / on-disk key registry stamps every entry with `kind`. Issuance time (`RegisterCollabKey` / `RegisterUpstreamKey`) populates `kind` from the issuance operation; revocation operates on the `(kind, keyValue)` tuple, not on `keyValue` alone (a `ck-ظخ` and a `uk-ظخ` are different keys even if their suffixes happened to collide).
- **`KeyList` server broadcast** ظ¤ when the server pushes the key registry to the dashboard (and to main for management), every entry in the list carries `kind` alongside `label` / `issuedAt` / `status`. The dashboard's key-management modal renders the list with a `kind`-badge per entry (visual differentiation: collab / upstream / local-key).
- **`HELLO` envelope `meta`** ظ¤ when an agent connects via a key-bearing URL, the server's `SERVER_HELLO` echo (┬د2 handshake) carries `meta.upstreamKey` / `meta.collabKey` / `meta.localKey` per the connection's key kind. The agent's HELLO response MAY echo the resolved `kind` for self-classification. The `meta` field is the canonical channel for the server to communicate the resolved authentication outcome to the agent without leaking the raw key value past the connection establishment point.
- **Dashboard key-management modal** ظ¤ the UI surface for issuing / revoking keys, listing live keys, and copying join URLs. The modal renders one tab or section per `kind` (or one consolidated list with `kind`-badge filtering), so the operator can distinguish collab keys (external collaborators) from upstream keys (peer agents) from local-keys (cross-environment local agents) at a glance. Issuance forms branch on `kind` to expose the right metadata (e.g., collab keys take a `label` + optional expiry; upstream keys take a `peerName` + capability hints).

**Composition**:

- **With ┬د2 connection paths** ظ¤ the `kind` enum maps onto the ┬د2 WS URL parameter (`?key=<ck-ظخ>` ظْ `kind:collab`; `?upstreamKey=<uk-ظخ>` ظْ `kind:upstream`; no key ظْ no `kind` ظْ default `local` role for the no-auth path). The `kind` does not replace the URL parameter; it is the *classification of the key value*, not the transport parameter.
- **With ┬د13.16.9 routing** ظ¤ key management envelopes (`RegisterCollabKey` / `RegisterUpstreamKey` / `KeyList` / `RevokeCollabKey`) are board-directed by intent (they target the dashboard's key-management surface and the server's key registry, not a peer agent's A2A inbox). The ┬د13.16.9 board-directed allowlist SHOULD include the key-management names; the v2.5.1 KEY-MGMT contract makes this an explicit composition rather than an implicit name-by-name discovery.
- **With ┬د13.14 redaction discipline** ظ¤ key values (`ck-ظخ` / `uk-ظخ` suffixes) are environment-specific credentials and MUST NOT appear in any public-surface artifact. The `kind` enum value itself is a vocabulary token (not a secret); only the key value is sensitive. Public reference docs MAY say "the server issues `kind:upstream` keys"; they MUST NOT say "the server issued `uk-9f3a2b1cظخ`".

**Scope**: contract clarification. The `RegisterCollabKey` / `RegisterUpstreamKey` / `CollabKeyIssued` / `KeyList` / `RevokeCollabKey` wire shapes (┬د5) gain an additional `kind` field at the top level (server-stamped on issuance, mandatory on `KeyList` entries, optional on `HELLO.meta`); existing adopters that omit `kind` on outbound continue to work for the legacy single-kind paths (collab via `RegisterCollabKey` was historically the only kind exercised), but the dashboard modal and any cross-kind operations require the field. Adopters SHOULD migrate to the `kind`-enum form in lockstep with the ┬د13.16.9 routing decision tree (v2.4.17 + v2.5.1) so the board-directed key-management envelopes route correctly to the dashboard's key-management surface and not into the main A2A inbox.

### 13.16.12 Dashboard render patterns ظ¤ interaction, date-line, attachment, connection, upstream visibility

**The contract**: six dashboard-render patterns codified as tier-1 SSoT discipline, distilled from the cycle_b track-2 reference consolidation (six newer patterns surfaced through a single hub multi-fix commit on 2026-06-01, in addition to the five prior patterns already captured in `constellation/reference/docs/dashboard-render-patterns.md` v2.5.1). The patterns are dashboard-side instances of the wire-level discipline already codified in ┬د13.11.3 (envelope normalization), ┬د13.16.9 (4-group routing classification), and ┬د13.13 (3-tier ack mapping); this sub-section consolidates them into a single named SSoT so downstream Constellation adopters can implement the same UX-coupled behavior without re-deriving each pattern from upstream commit history.

**Why this sub-section, not "just read dashboard-render-patterns.md"**: the reference doc is the *render-tier companion* (renderer implementer's manual ظ¤ file paths, function names, CSS selectors), while this ┬د13.16.12 is the *protocol-tier abstraction* (the *invariant* a renderer must satisfy regardless of stack). An adopter building a non-vanilla dashboard (React, Vue, Svelte, native) reads ┬د13.16.12 for the contract and the reference doc for the vanilla-DOM concrete implementation.

**Pattern 1 ظ¤ Collapse indicator: glyph unification + CSS rotation discipline**

The collapse-state indicator (expanded vs collapsed) MUST use a **single glyph rotated by CSS**, not two distinct glyphs swapped at toggle time. Rationale: glyph-pair patterns (e.g., `ظû╛ expanded / ظù collapsed`) suffer from per-glyph metric mismatch ظ¤ the bounding box, baseline alignment, and visual weight differ between the two characters, producing a 1-2 px visual jitter on every toggle that operators report as "the row twitches". The fixed-glyph + `transform: rotate()` pattern preserves the same bounding box on both states and lets the browser animate the transition smoothly. Canonical implementation: `textContent = "ظù"` constant; `.expanded { transform: rotate(-90deg); transition: transform .15s }`. Adopters MUST NOT use the `ظû╛ ظù` swap pattern even though it appears simpler ظ¤ the metric mismatch is observable. Composition: the `[hidden]` CSS pitfall (`dashboard-render-patterns.md ┬د3` ظ¤ `display: flex` overriding the UA default `[hidden] { display: none }`) is orthogonal and still applies; the rotation pattern is the *indicator* discipline, the `[hidden]` fix is the *content visibility* discipline.

**Pattern 2 ظ¤ Date-line divider: client-tz day-boundary insertion + sticky positioning**

The chronological message stream MUST insert a date-line divider at every client-local day boundary, with the day computed from `row.timestamp` (epoch ms, client-side `new Date(epoch)`) ظ¤ NEVER from the wire-format ISO string (the wire string carries sender's tz; the divider must reflect the *viewer's* local-day boundary, not the sender's). The divider element uses `position: sticky; top: 0` so the most-recent day label remains visible during scroll; the sticky-top offset MUST account for the chat-panel header height (not the page header height ظ¤ a date-line floating behind the chat panel header is a UX bug). Visual canonical: centered day label (locale-dependent ظ¤ `هءجنèء / هû┤هب£ / YYYY-MM-DD` for Korean, `Today / Yesterday / YYYY-MM-DD` for English) with `~ ~` decorators on either side (rendered via `::before` / `::after` CSS pseudo-elements, not literal characters in the DOM ظ¤ keeps the divider's text content programmatically accessible as just the date). The scroll container (the message stream itself) MUST have `padding-top: 0` so the sticky divider docks tight to the top edge of its viewport. Composition: this pattern composes with ┬د13.11.3 rule 3 (timestamp obligation + client-local render) ظ¤ the server stamps `msg.timestamp = Date.now()` if absent (rule 3), and the dashboard uses that epoch for the divider's day computation (this pattern).

**Pattern 3 ظ¤ A2A attachment chip: mime detection + preview + graceful localPath**

A2A messages carrying attachments (the `value.attachments[]` / `value.zip` / `value.attachment` field-name choice is portable per `dashboard-render-patterns.md ┬د6`) MUST render the attachment metadata as an inline chip inside the message card body ظ¤ NOT as a separate row beneath the message. The chip composes: mime-icon glyph (emoji or icon font; `≡اô / ≡اû╝ي╕ / ≡اù£ي╕ / ≡ا╡` for text/image/archive/audio) + filename + size + (preview trigger | download trigger). Preview triggers MUST handle both wire shapes: (a) `dataUrl` or `url` field present ظْ inline modal renders the resource; (b) only `localPath` present (the attachment lives on the sender's filesystem, not transported in the envelope) ظْ chip still renders gracefully with the localPath as informational metadata + a "local file, not transported" indicator, NEVER an error or a broken-link icon. Download triggers MUST handle the same two cases (transport vs metadata-only). Rationale for "metadata-only graceful display": targetless / metadata-only attachments are a common emit pattern when the sender means "here's a file I have, name and check it" rather than "here's a file I'm transmitting"; treating the metadata-only case as a broken transport pollutes the dashboard with false-error indicators. Composition: ┬د13.13 application-tier ack ظ¤ an `AckProcessed` on an attachment message means the recipient agent has *surfaced* the metadata, NOT that the recipient has downloaded the localPath resource (which they cannot do without filesystem access to the sender).

**Pattern 4 ظ¤ Closed-session connection indicator: present-set based color dot**

Each agent row in the roster (sidebar / dropdown / agent list) MUST render a connection indicator dot whose color reflects the agent's *present membership* in the live `AgentList` from the server ظ¤ NOT the row's display-visibility state. Canonical color binding: present in `AgentList` ظْ green dot `ظù`; absent (history-only ظ¤ the row exists because the agent has sent messages but is not currently connected) ظْ gray dot `ظù`. The dot updates on every `AgentList` push from the server (┬د13.9 onboarding handshake group). Tooltip on the dot MUST surface the precise connection state (`connected since <local-tz timestamp>` / `last seen <local-tz timestamp>`) ظ¤ local-tz, NOT wire-format (composes with Pattern 2's client-local rendering discipline). Composition with ┬د13.13: a `delivered` ack (transport-tier) on a message to a gray-dot agent is wire suspect (the agent is not currently connected to receive); the dashboard MAY surface this with an inline warning ("delivered to inbox, recipient currently offline") rather than escalating silently to commitment-tier wait ظ¤ closes the "wait forever on a disconnected agent" gap.

**Pattern 5 ظ¤ Upstream tab re-exposure: present-set sweep, not just delta**

When the roster includes manually-hidden tabs (e.g., the operator collapsed an upstream agent's view because the conversation was finished), the auto-re-expose logic MUST trigger on **every `AgentList` push where the agent is present**, not only on the `absent ظْ present` delta. Rationale: an upstream agent that stays continuously connected (long-lived peer) never produces an `absent ظْ present` delta after initial join; if the operator collapses the tab mid-conversation, the tab stays collapsed forever even though the agent is actively producing messages. The corrected discipline: on every `AgentList`, for each `present` agent, if the agent's tab is currently `hidden`, un-hide it (the next message will re-attract the operator's attention without the operator having to manually re-expose the tab). The `present` sweep MUST be idempotent ظ¤ re-exposing an already-visible tab is a no-op, not an error. Composition: this is the "long-lived upstream peer" failure mode that the ┬د13.16.9 handshake-group classification (v2.5.2) does not catch ظ¤ `AgentList` is in the handshake group, so the watcher correctly does NOT wake on each push, but the dashboard MUST still re-evaluate tab visibility on each push regardless of watcher-wake intent.

**Pattern 6 ظ¤ Upstream A2A classification (receiver-side): source-set-agnostic detection + text relay fallback**

The dashboard's receiver-side A2A-intent detection MUST use the negation form of ┬د13.11.3 rule 2: `source !== "user" && source !== "board" && targetAgentId && targetAgentId !== self`. CRITICALLY, the detection MUST NOT use the positive-set form `source === "agent"` because upstream peer agents emit `source = <their-agentId>` (a nonstandard but conformant source value); the positive-set check silently false-negatives ALL upstream A2A traffic, producing the operator complaint "upstream A2A is invisible on dashboard". The negation form is portable across the standard `source: "agent"` and the upstream variant `source: <agentId>` and any future emitter variant. **Text relay fallback** (the second half of this pattern): the server's outbound routing (┬د13.16.9 routing decision tree) MUST relay target-unspecified TEXT_MESSAGE envelopes to `wsPrimary` (the main A2A path) when the target-unspecified TEXT_MESSAGE is A2A-intent ظ¤ i.e., the routing decision tree's fail-safe-default-A2A-intent branch (the fourth branch of the tree) MUST cover TEXT_MESSAGE name too, not just CUSTOM. Without this, upstream agents that emit raw text (without wrapping in CUSTOM) silently drop on the server-side normalization step. Composition: ┬د13.11.3 rule 1 (channel `type` normalization ظ¤ server stamps `type: "CUSTOM"` when absent) handles the producer-side defect; this pattern handles the *router*-side defect (which is independent of the type stamp).

**Pattern 7 ظ¤ Board render TEXT_MESSAGE fallback to A2A card (defense layer for non-conformant adopters)**

The receiver-side dashboard MAY detect `TEXT_MESSAGE` envelopes whose `text` field carries a serialized A2A Report (typically as a `[A2A Report]` lead line followed by a fenced JSON code block, e.g., ```` ```json ```` containing `{ type: "Report", ... }`) and **promote** the message into an A2A card render (the `a2acard` surface that ┬د13.16.12 Pattern 6 + Pattern 4 already define for native A2A-intent envelopes). The promotion extracts the JSON payload from the code-block, parses it, and renders the resulting object as if it had arrived as a native `CUSTOM` envelope: name/agentId/msgId/threadId in the envelope summary card; the parsed body as the collapsible JSON body; attachment chips (┬د13.11 rule 5 / Pattern 3) if `attachments[]` is parseable from the JSON.

**This is a defense-layer pattern, NOT a substitute for the producer-side fix.** The canonical solution is the sender's gateway-client emitting **structured envelopes per ┬د13.11 + ┬د13.11.3 rules 1-5** (named `CUSTOM` with `value` + optional `attachments[]`, not a TEXT_MESSAGE with the same content serialized into `text`). When the sender is a non-conformant adopter ظ¤ typically a partial reimplementation of the ┬د13.9 / ┬د13.11 contract that has not yet wired its outbound emit path to produce structured envelopes ظ¤ the receiver's board would otherwise display the serialized JSON as a raw text bubble, losing the A2A-intent surface (no card, no attachment chips, no ┬د13.11 rule 5 mime / dataUrl handling, no ┬د13.13 ack-tier indicators on the rendered surface). Pattern 7 gives the receiver a way to **recover the A2A-intent surface from the serialized text** while the producer-side fix is in flight, preventing the missing-card defect from reaching the operator.

**Implementation discipline**:

1. The detection MUST be a `TEXT_MESSAGE` branch (not a `CUSTOM` branch ظ¤ the canonical `CUSTOM`-via-┬د13.11 path is unchanged). The branch runs *after* the normal TEXT_MESSAGE render is computed but *before* it commits to the DOM; if the extraction succeeds, the rendered surface swaps to the A2A card form. If the extraction fails (no `[A2A Report]` lead, no fenced JSON, JSON parse error), the message renders as a normal TEXT_MESSAGE (the fallback gracefully degrades; never errors loudly).
2. The promoted card MUST be visually marked as **fallback-promoted** (e.g., a small `fallback` badge on the card header, or a tooltip "promoted from TEXT_MESSAGE") so the operator can distinguish a canonical `CUSTOM`-rendered card from a Pattern-7-promoted card. The marking serves two purposes: it surfaces *which adopters are still non-conformant* (the operator sees the badge on every Pattern-7 hit), and it preserves audit-trail integrity (the original wire frame was TEXT_MESSAGE, not CUSTOM ظ¤ the render-tier promotion does not rewrite history).
3. The promotion MUST NOT alter the outbound ack path. Transport-tier `Ack` continues to be the server's mechanic on the original TEXT_MESSAGE frame; commitment-tier `AckProcessed` (if emitted) follows ┬د13.13 normally; the promotion is *render-layer only*.
4. The promotion MUST extract `attachments[]` from the JSON when present, but the chips render against the ┬د13.11 rule 5 graceful-localPath discipline (Pattern 3) ظ¤ a Pattern-7-promoted card with `source: "embedded"` attachments handles the dataUrl correctly (bytes were in the TEXT_MESSAGE.text serialization); a Pattern-7-promoted card with `source: "local_path"` attachments still renders the chip with the "local file, not transported" indicator (no error). This composes Pattern 7 with Pattern 3 at the chip layer: Pattern 7 supplies the parsed attachment array; Pattern 3 supplies the per-chip render discipline.

### 13.17 Main-chat structured-choice prompts are FORBIDDEN ظ¤ route via the live board

**Discipline** (not a rule of nature): in a Constellation main-chat window, the agent MUST NOT emit an `AskUserQuestion`-style structured choice prompt (numbered options, multi-select panels, picklists) inline in the chat stream. Free-form text questions ("which framework do you want?") are fine ظ¤ the prohibition is specifically on **structured option UIs** rendered into the main-chat reply.

**Why** (the reasoning, since this is a contract not a primitive):

- **Main-chat is point-to-one, agent-to-user.** A structured choice rendered inline produces a UX-coupled artifact (option panel) that lives only inside that one chat scrollback. There is no board-side projection ظ¤ the question, the options, and the user's pick are all invisible to other agents, to the watcher's replay history, and to the cross-agent SSoT (┬د3 board = visualization SSoT).
- **Structured Q/A needs board mediation for traceability.** A board-routed question writes to `state.json` + replay history with a stable `msgId` (┬د13.13), so the question and its resolution are part of the auditable A2A history ظ¤ re-readable across turns, across agents, and across restarts. Inline main-chat options leave no such trace.
- **Multi-agent visibility.** When the main agent is asking the user something, sibling/worker agents on the same board need to know the main is *parked on a user input* (so they don't issue Delegates expecting an immediate response, and so the board reflects the actual blocked state). An inline option UI hides this ظ¤ the board reads "main online, idle" while the main is actually awaiting a structured pick.
- **The free-form text exception**: ordinary prose questions in main-chat are fine because they don't promise a structured artifact ظ¤ they're conversational, and the response shape is text the agent then summarizes back to the board. Structured option UIs *do* promise a structured artifact, and that artifact's natural home is the board, not the chat scrollback.

**Two escalation paths** (per main repo upstream policy #414):

1. **defer-OK** ظ¤ route the question into the live-board **review-items tab**. The user picks it up at their own cadence; the agent re-arms its watcher (┬د13.16.6) and waits. This is the default path; use it whenever the question is not blocking the immediate next-step turn.
2. **need-now** ظ¤ route to the **real-time chat window with options UI in the live board** (new UI6 ظ¤ see task B in the originating delegate). This carries the same structured-pick semantics that an inline `AskUserQuestion` would have, but on the board side so it gets traceability + multi-agent visibility automatically.

**Transitional note**: the need-now UI6 (live-board real-time chat + options) is pending main board features. Until UI6 ships, the recommended fallback is *defer-OK* (review-items tab) plus a one-line WorkerReport to the board noting "user input pending, defer-OK path". Do not regress to inline `AskUserQuestion` while waiting for UI6 ظ¤ the transitional fallback is board-mediated, just slower.

**Failure mode this prevents**: a main agent that emits an `AskUserQuestion` inline in main-chat creates a question that no other agent can see, no replay can reconstruct, and no audit can verify. If the user closes the chat without picking, the question is lost; if the user picks, the pick exists only in the chat history and the main's working memory. Board mediation makes the question a first-class A2A artifact.

**Review-items tab badge discipline** (board UI sub-rule for the defer-OK path above):

1. **Tab visibility ظ¤ always shown**. The review-items tab (the `decisions` panel) is a **constant tab** in the live-board navigation; it does NOT appear/disappear based on whether items exist. The tab is rendered regardless of count ظ¤ even when the queue is empty (count = 0).
2. **Badge counter ظ¤ count > 0 only**. The badge element (a small decoration on the tab carrying the integer count) is shown when `count > 0` and **hidden when `count === 0`**. Implementation discipline: `app.js` sets the badge element's `hidden` attribute based on the live count (`badge.hidden = (count === 0)`); `style.css` enforces the visual hide via `#decisions-badge[hidden] { display: none; }` (i.e. the standard `[hidden]` selector pattern, not a separate `.empty` class). Tab stays visible; only the badge decoration disappears.
3. **Rationale**. Keeping the tab visible signals to the user that the escalation path **exists and is available** even when the queue is empty ظ¤ this is what encourages defer-OK use over inline `AskUserQuestion` attempts (if the tab disappeared when empty, the path would feel "off" rather than "ready"). Hiding the badge avoids zero-noise visual clutter: a `0` badge would either look like an alert (false positive) or like dead UI chrome (visual noise without information).
4. **Future board reimplementations** MUST preserve both halves: constant-tab + count-gated badge. Re-implementing the panel as "hide the tab when empty" (the lazy alternative) regresses the ┬د13.17 affordance ظ¤ the user loses the visible signal that the defer-OK path is available, which raises the felt cost of routing structured questions through the board and pulls agents back toward inline `AskUserQuestion` violations.

### 13.18 Non-branching choices ظ¤ recommend + proceed (no escalation)

**Discipline**: when EG has a clear recommendation and the choice is **not a fork-of-architecture decision**, EG announces the recommendation in **1 line** and **proceeds**. User pivot remains possible at any time, but is **not solicited** ظ¤ soliciting it via a structured option list (A/B/C/D?) after the recommendation is itself the violation this rule prevents.

**Scope** ظ¤ this rule fires on **non-branching choices**, defined as:

- **Priority ordering** of already-decided tasks (which of the agreed deliverables ships first).
- **Sequential-vs-parallel** of already-decided tasks (whether to dispatch in serial lanes or Superscalar lanes ظ¤ both produce the same final artifacts).
- **Selection between near-equivalent options** (small parameter choices, file-organization minutiae, wording variants, when one option is clearly recommended and the alternatives don't change the *architecture* of the outcome).

It does **not** fire on **fork-of-architecture** decisions ظ¤ those still go through ┬د13.17's escalation paths (defer-OK ظْ review-items tab; need-now ظْ live-board real-time chat + options UI, pending UI6). A fork-of-architecture decision is one where the alternatives produce *materially different* downstream artifacts (different repo layouts, different protocols, different schemas) such that the user's pick reshapes the work, not just its ordering.

**Parallelism default presumption**: when parallelism is feasible (Superscalar `issue_width > 1` available, lanes can be made independent, cost-benefit gate clears per Principle #15), **parallel is the default presumption** absent explicit user steering. EG does not ask "should I parallelize this?" ظ¤ EG announces "running in N lanes, retire in declared order" and proceeds. The user pivots if they want serial; the agent does not solicit.

**Failure mode this prevents** (the originating incident ظ¤ main upstream policy #411 / Delegate seq 85): the agent identified option D as the recommendation, then asked "A/B/C/D?" anyway. That sequence wasted a user round-trip ظ¤ the recommendation was already the answer, and the structured-choice ask treated a non-fork ordering decision as if it were a fork-of-architecture decision. The pattern to avoid: *recommendation present ظْ option list emitted ظْ user round-trip ظْ user picks the recommendation*. The correct pattern: *recommendation present ظْ 1-line announce ظْ proceed ظْ user pivots only if they disagree*.

**Why the 1-line announce still matters**: silence isn't right either ظ¤ the user needs visibility into what's happening so they *can* pivot if they want to. The 1-line announce is the minimum information transfer (what's being done, why, expected next state) that preserves the pivot option without forcing a round-trip. It's a *broadcast*, not a *solicitation*.

**Cross-link**:

- **┬د13.17** (main-chat structured-choice prompts FORBIDDEN) ظ¤ that section bans the *form* (inline AskUserQuestion-style option UIs in main-chat); this section bans the *trigger* (asking at all when the choice is non-branching and a recommendation exists). ┬د13.17's escalation paths (defer-OK / need-now via board) apply only when the question is *legitimately needed* ظ¤ ┬د13.18 is the upstream filter that determines whether the question is legitimately needed in the first place.
- **Seed Principle #16** (autonomous execution ظ¤ absolute): the seed's existing gate set (loss/external-publish ┬╖ new major-fork decision ┬╖ restart-deploy timing ┬╖ explicit user steering) already implies non-branching choices proceed without escalation, but the originating incident shows the implication wasn't operative. ┬د13.18 is the explicit operational form of Principle #16 for the *choice-emission* surface; Principle #16's strengthened sentence ("recommendation present + non-branching = proceed without escalating; parallelism is the default presumption when feasible") is the seed-side mirror of this section.
- **┬د13.21** (Fresh-context defer is an anti-pattern): ┬د13.18's "recommend + proceed" is the *choice-surface* discipline for non-branching forks; ┬د13.21 is the *time-surface* discipline for the same autonomy principle ظ¤ once the recommendation is announced and proceed-mode entered, the agent does NOT silently re-route the work into "next fresh context will pick this up". The policy chain is **┬د13.18 recommend ظْ ┬د13.16.6 element 5 planned-queue scan (#442) ظْ ┬د13.21 anti-defer**: the recommendation identifies the work, the queue scan picks the next item *now*, and ┬د13.21 forbids the deferral that would otherwise quietly undo both. ### 13.19 Deadlock resolution ظ¤ formal spec (prevention ┬╖ detection ┬╖ auto-resolution ┬╖ escalation, layered above ┬د13.13)

#### 13.19.2 Wait-pattern classification ظ¤ strict / quasi / healthy (livelock = quasi variant)

A2A wait situations are not a single phenomenon ظ¤ the appropriate resolution depends on which class is in play. Three named classes plus one explicitly subsumed variant:

- **Strict deadlock** ظ¤ a cycle in the A2A message-dependency graph where neither party can make forward progress without the other releasing first. The wait is *structural*; no amount of patience or SLA-renegotiation resolves it. Detection: cycle in the wait-edge DFS (┬د13.19.3). Resolution tier: strong primitives ظ¤ preemption (┬د13.19.5) or escalation (┬د13.19.6). Rare in the current Constellation topology because the ┬د13.9 role split (workers don't issue `Delegate` back) minimizes cycle topology.
- **Quasi-deadlock** ظ¤ a wait where one party COULD progress immediately by completing a deferred action, but defers across cycles. The wait is *behavioral*, not structural. Detection: a wait edge older than the agreed `ReviewSLAAck` budget with no cycle. Resolution tier: SLA-ack discipline (┬د13.19.4) + priority re-ordering (┬د13.19.6 tier 1). Most common class in the current EGظ¤main deployment ظ¤ this is what the 2026-05-31 deferred-review incident actually was.
- **Healthy wait** ظ¤ a wait of bounded duration where the counterparty is making expected progress (worker on `Delegate` standby per ┬د13.11.2; A2A sender waiting on a recipient currently in a long-running foreground turn). Detection: bounded by ┬د13.13 `Ping`/`Pong` keepalive and ┬د13.16.6 `WAKE`/`REARM` markers ظ¤ the recipient turn is active. Resolution tier: **none ظ¤ the protocol is functioning**. Misclassifying healthy wait as deadlock is itself a failure mode (alarm-fatigue per ┬د13.13).
- **Livelock ظ¤ explicitly classified as a quasi-deadlock variant** (consensus diff_adjusted_5 #2): mutual deferral where both parties repeatedly *attempt* to yield to the other (e.g. each emits `PreemptRequest` against the other in alternating turns, with each release immediately re-blocked by the other side's symmetrical request). Livelock has no structural cycle by the strict-graph definition (each party is technically making local "progress" by issuing yields), but its wall-clock behavior is identical to quasi-deadlock (deferred work fails to retire). Detection and resolution route through the quasi-deadlock path: SLA-bound the yield-request count, escalate to mediation/leader-override (┬د13.19.6 tier 2/3) before the yield-spiral indefinitely consumes turns.

Misclassification cost is asymmetric: treating *strict* as *quasi* loses time (SLA renegotiation cannot close a structural cycle); treating *quasi* as *strict* burns alarm-fatigue budget (preemption/escalation primitives fire against a wait that ordinary commitment discipline would have resolved); treating either as *healthy* is the silent-degrade class that the ┬د13.13 conservative-multi-probe + this section's SLA discipline exist to prevent.

#### 13.19.3 Detection ظ¤ wait-edge DFS (2-agent sufficient) + wait-class timeouts + ┬د13.16.6 agent-active correction

Detection layer composes three signals; each is necessary, none is sufficient alone.

**(1) Wait-edge graph DFS ظ¤ 2-agent depth sufficient** (consensus diff_adjusted_5 #1). The Constellation A2A wire already carries cycle-detection metadata: every `CUSTOM` carries a sender-bridge-stamped `msgId` (┬د13.13); replies carry `re_msgId` (or `parentId`); an agent in `WAITING` for `re_msgId = X` declares a wait edge on the sender of `X`. At each agent's turn-start (and at the watcher's rearm path), run a depth-limited DFS over the local `ws-history/<agentId>.jsonl` (┬د2 / ┬د13.16.3) restricted to **wait-edge depth ظëج 2**, identifying whether the local agent participates in a 2-cycle (A waits on B, B waits on A). The edge-chasing N-agent extension (Chandy-Misra-Haas / Mitchell-Merritt path-pushing across arbitrary depth) is **explicitly deferred** ظ¤ for the current Constellation topology (typically ظëج ~10 active peers, with the ┬د13.9 orchestration tree pinning most edges to non-cycle shapes), 2-agent DFS is sufficient. The N-extension is a future-work hook (deploy when a 3-or-more cycle is empirically observed in dogfood), not part of this spec's required behavior.

**(2) Wait-class timeout** ظ¤ per ┬د13.19.2 class:

| Wait class | Timeout source | Notes |
|------------|----------------|-------|
| Healthy (Delegate-standby) | unbounded | ┬د13.9 / ┬د13.11.2 ظ¤ no autonomous heartbeat fires while parked |
| Healthy (in-turn recipient) | one `Ping`/`Pong` round-trip + ┬د13.16.6 agent-active correction | ┬د13.13 keepalive |
| Quasi-deadlock candidate | review-SLA budget (negotiated per A2A message; default ~24h / 2 cycles, set by `ReviewSLAAck.eta`) | ┬د13.19.4 |
| Strict-deadlock candidate | 2├ù longest healthy-wait timeout in the cycle | ┬د13.19.5 strong-primitive tier |

**(3) ┬د13.16.6 agent-active correction ظ¤ mandatory** (consensus, applies to every timeout above). The harness `SIGSTOP`s background tasks while the agent's foreground turn is active and `SIGCONT`s them when the turn ends. A naive wall-clock timeout therefore overcounts: a 60 min wall-clock window with 25 min of foreground turn activity has true watcher-elapsed of ~35 min. The formal expression (carried over from ┬د13.16.6, repeated here for the detection rule):

```
effective_elapsed = wall_clock_elapsed ظêْ ╬ث agent_active_duration (during wait window)
```

All ┬د13.19.3 timeout decisions use `effective_elapsed`, never raw wall-clock. The agent-active duration is observable via the watcher's `WAKE`/`REARM` markers (┬د13.16.6), the recipient's `ws-history` emission timestamps, or (if added) an explicit `CUSTOM/TurnActive`/`TurnEnded` pair on the wire ظ¤ the choice between observable sources is implementation-level, not spec-level. What is spec-level: the timeout MUST be corrected; an uncorrected wall-clock comparator is non-compliant with ┬د13.19.

**Termination signals as detection input** (consensus diff_adjusted_5 #5 ظ¤ mandatory). Three termination signals MUST be carried in the wire vocabulary as `CUSTOM` payloads:

- `DONE { for: msgId, summary? }` ظ¤ recipient completed the work the inbound implies; equivalent to `AckProcessed` at the application layer (┬د13.13) but with explicit "no further action expected" semantics.
- `BLOCKED { for: msgId, reason, waitingOn: msgId | external-dependency-id }` ظ¤ recipient cannot progress *this turn*; the `waitingOn` field is **mandatory and IS the wait-edge graph input** for ┬د13.19.3 DFS. An agent that emits `BLOCKED` without `waitingOn` produces an undetectable wait ظ¤ non-compliant.
- `NEEDS_HUMAN { for: msgId, question | summary }` ظ¤ recipient cannot decide autonomously; escalates immediately to the ┬د13.19.6 tier-4 path (via ┬د13.17 `decisions` panel).

#### 13.19.4 Prevention layer ظ¤ priority ordering + ReviewSLAAck + termination signals

Prevention fires *before* a wait becomes either quasi-deadlock or strict cycle. Three rules:

**(A) Priority ordering via ┬د13.9 orchestration tree** ظ¤ `main > collab > upstream > local`. When two agents could each plausibly defer to the other, the lower-priority party yields *by default* without negotiation. This is cheap (no extra wire traffic), non-coercive (both parties agree on ┬د13.9 as the shared SSoT), and prevents most quasi-deadlocks before they form. The orchestration tree is a *prevention* control, not just a resolution control ظ¤ the lower-priority party emitting their work first removes the wait-edge before it exists.

**(B) `ReviewSLAAck { ackFor: msgId, eta, kind: 'SLA-OR-WORK', note? }` between `Ack{delivered}` and `AckProcessed`** ظ¤ the commitment-tier ack (┬د13.19.7). On A2A inbound that implies a *deferred response* (review request, decision request, opinion request ظ¤ anything not "immediately processed and acked"), the recipient agent MUST emit `ReviewSLAAck` as part of its inbound processing, **before** entering the deferred-work cycle. `eta` is either ISO-Z wall-clock or `cycles: N`. The `SLA-OR-WORK` semantics: *"I will either complete this work within `eta`, OR if I cannot, I will emit a renegotiate-SLA message BEFORE `eta` expires explaining the new ETA and reason. I will NOT silently let the SLA elapse."* Silent expiry is a ┬د13.19.6 escalation trigger; renegotiation before expiry is normal (subject to the ┬د13.19.10 Q6 frequency limit).

**(C) Termination signals are emission-mandatory** (per ┬د13.19.3) ظ¤ `DONE` / `BLOCKED{reason, waitingOn}` / `NEEDS_HUMAN`. An agent that processes an inbound and produces *no* termination signal leaves the wait edge ambiguous (is the recipient still working? is it blocked? has it silently finished?). Ambiguity is the precondition of every wait-class misclassification in ┬د13.19.2; mandatory termination signals remove the ambiguity at the source.

Prevention is the *cheap* layer ظ¤ none of (A)/(B)/(C) requires a primitive emission beyond what the agent would emit anyway as part of inbound handling. The expensive layers (detection, auto-resolution, escalation) fire only when prevention fails.

#### 13.19.5 Auto-resolution primitives ظ¤ preemption + deadline + asymmetric leader + starvation guard

When prevention fails and detection (┬د13.19.3) flags a candidate wait class, four primitives apply *autonomously* (no human in the loop) before ┬د13.19.6 escalation:

**(1) Cooperative preemption ظ¤ `PreemptRequest { releasing: msgId, reason }`** ظ¤ the lower-priority party voluntarily releases the wait edge, returning to standby. The higher-priority party proceeds; the released item is retried after the unblocking work completes. This is the symmetric, cheap form ظ¤ both parties agree on the ┬د13.9 tree, so the lower-priority emission is uncontested.

**(2) Asymmetric leader override ظ¤ `PreemptForce { reclaiming: msgId, reason }`** ظ¤ the higher-priority party emits a forced reclaim. The lower-priority party MUST release; the live-board logs the forced reclaim for human audit (`decisions` panel persistence, ┬د13.13). `PreemptForce` is the *strong* form and should be rare ظ¤ its use implies the lower-priority party refused to yield cooperatively, which is itself a discipline failure surfaced to the human watching the board. The asymmetry follows ┬د13.9: a `PreemptForce` *up* the tree (lower-priority issuing against higher) is non-compliant; the wire-level filter (server-side, per ┬د13.19.8 permission table) rejects it.

**(3) Deadline enforcement** ظ¤ when `ReviewSLAAck.eta` elapses (corrected per ┬د13.16.6) WITHOUT either a `DONE` / `BLOCKED` / `NEEDS_HUMAN` from the recipient OR a renegotiate-SLA, the sender is authorized to invoke (1) or (2) per ┬د13.9 priority. The deadline is *enforced* in the sense that elapsed-without-renegotiation IS a ┬د13.19.2 quasi-deadlock signal ظ¤ the sender does not need additional evidence; the SLA contract was the evidence.

**(4) Starvation guard ظ¤ Mitchell-Merritt-style priority bump** (consensus diff_adjusted_5 #4 ظ¤ accepted from main RRP). When the same low-priority party has had its `PreemptRequest` denied (or its `ReviewSLAAck` renegotiated by a higher-priority counterparty) **N times in succession** (default `N = 3`), the low-priority party's effective priority is bumped one tier for the duration of the contested wait. This prevents the lowest-priority agent from being indefinitely starved by a chain of higher-priority preemptions ظ¤ the classic priority-inversion failure mode. The bump is *contest-local* (only the specific wait edge is bumped, not the agent's global tier) and *transient* (resets when the contested wait clears). Implementation: the sender tracks the denial count in its outbox state; on the Nth denial, the next emission carries `priorityBump: true` and the receiving side treats it at the bumped tier for the auto-resolution decision. The default `N = 3` is calibrated against the ┬د13.17 human-escalation cognitive-bandwidth ceiling (┬د13.19.10 Q4) ظ¤ three denials is enough to surface a genuine starvation case without firing on transient mutual deferrals that would have cleared anyway.

#### 13.19.6 Escalation ظ¤ 4-tier ladder (ordering ظْ preemption ظْ mediation ظْ human via ┬د13.17 decisions panel)

When prevention fails and the ┬د13.19.5 primitives do not break the wait within their budgets, escalation proceeds in **strict tier order** (no skipping; each tier must be attempted before the next). This is itself a ┬د13.18 non-branching choice path ظ¤ the agent does not ask the human "which tier?" ظ¤ it advances mechanically.

| Tier | Primitive | Trigger condition | Cost |
|------|-----------|-------------------|------|
| 1 | **Priority ordering** (┬د13.19.4 A re-applied as resolution) | Quasi-deadlock detected; both parties still operating per ┬د13.9 tree | Free ظ¤ emit nothing new; lower-priority resumes its deferred work |
| 2 | **Preemption** (┬د13.19.5 (1) cooperative, (2) forced if needed) | Tier 1 doesn't break the wait within one rearm interval | One `PreemptRequest` or `PreemptForce` emission |
| 3 | **Mediation** ظ¤ `MediationProposal { cycleMembers[], proposedReleaseOrder[] }` | Tier 2 fails or peer-to-peer cycle (collab ظ¤ collab) where neither party is higher-priority | One mediator emission + acks from cycle members (`MediationAck`) |
| 4 | **Human escalation via ┬د13.17** ظ¤ `EscalationRequest { cycle, attemptedPrimitives[], summary }` to board `decisions` panel | Tier 3 fails OR `NEEDS_HUMAN` termination signal observed at any earlier tier | Human attention ظ¤ the scarcest resource (alarm-fatigue gate, see ┬د13.19.10 Q4) |

Tier 4 routes via ┬د13.17 (live-board `decisions` panel, NOT inline `AskUserQuestion` in main-chat). The escalation becomes a first-class A2A artifact re-readable across turns; the human picks a release order (or restructures the dependency to remove the cycle) and emits a `decisions`-panel resolution that affected agents pick up via `UserPrompt`.

#### 13.19.7 3-tier ack layering ظ¤ transport / commitment / application

┬د13.13 specifies two principal ack grades (`Ack{delivered}` server-auto + `AckProcessed` agent-emitted). ┬د13.19 interpolates a third ظ¤ `ReviewSLAAck` ظ¤ between them, producing a clean three-tier layering:

| Tier | Ack | Emitter | Meaning | When |
|------|-----|---------|---------|------|
| 1 (transport) | `Ack { ackFor, kind: 'delivered' }` | **Server** (auto) | "The wire carried it." | Immediately on relay (┬د13.13). NOT itself acked; NOT displayed on the board (alarm-fatigue). |
| 2 (commitment) | `ReviewSLAAck { ackFor, eta, kind: 'SLA-OR-WORK' }` | **Recipient agent** | "I will respond within `eta`, OR I will renegotiate before expiry." | On inbound parse for deferred-response messages (┬د13.19.4 B). Mandatory for messages whose response is not synchronous with inbound handling. |
| 3 (application) | `AckProcessed { ackFor, kind: 'processed', summary? }` | **Recipient agent** | "I have acted on the inbound; the work is complete." | On completion of the implied work (┬د13.13). The WILCO; conflating with `delivered` (or with `ReviewSLAAck`) is a named failure mode. |

The tier-2 interpolation closes the gap that originated the 2026-05-31 incident: under the ┬د13.13-only layering, a sender saw `delivered` (server) and then *nothing* until `AckProcessed` (recipient) ظ¤ silent deferral across cycles fit inside that gap. The tier-2 ack makes deferral *explicit and SLA-bounded*, converting silent-degrade into either contract-honored (no friction) or contract-renegotiated (visible reason) or contract-breached (┬د13.19.6 escalation trigger).

**Bridge MUST NOT auto-emit `ReviewSLAAck`** ظ¤ same rationale as the ┬د13.13 no-auto-pong rule. A bridge that auto-acks at tier 2 would collapse the commitment signal that the whole ┬د13.19 mechanism depends on. `ReviewSLAAck` is *only* the recipient agent's emission, from an active runtime turn; from anywhere else it is a false commitment that downstream layers will misread.

#### 13.19.10 Q2 / Q3 / Q4 / Q6 decisions

The EG v0.1 draft enumerated six open questions for joint reconciliation; Q1 (edge-chasing N-extension) was answered "deferred" by consensus diff_adjusted_5 #1 and Q5 (turn-based vs continuous-process model) is absorbed by ┬د13.19.3 agent-active correction. The remaining four are decided here as part of the formal spec (not deferred to deployment-time policy):

**Q2 ظ¤ CNP (Contract Net Protocol) fit: REJECTED as wire vocabulary; PRESERVED as discipline borrowing.** The v0.1 draft observed that ┬د13.19.4's `ReviewSLAAck` borrows CNP's *spirit* (announce ظْ bid ظْ award ظْ result). The formal CNP protocol's *bid* phase, however, has no obvious Constellation primitive ظ¤ CNP supports multi-bidder negotiation (`announce ظْ many bids ظْ one award`), whereas the ┬د13.9 orchestration tree pins recipients to specific peers (a `Delegate` to main is not a CNP `announce` open to all peers). **Decision**: do NOT add a CNP `bid` primitive to the wire vocabulary. The bilateral SLA pattern (`ReviewSLAAck` from a single named recipient) is the only commitment-tier ack ┬د13.19 specifies; multi-bidder review can be modeled at the application layer (PM dispatches separate `Delegate`s to each candidate worker and selects from `WorkerReport`s) without needing CNP semantics on the A2A wire. **Reasoning**: minimal wire vocabulary is itself a ┬د13.13 invariant ("envelope convention + ack layer is one coherent family"); adding a `bid` type to support a deployment shape that does not currently exist would violate the YAGNI gate that ┬د13.13 took pains to establish.

**Q3 ظ¤ Mediator placement: HYBRID ظ¤ board adapter as default, third-party mediator agent as opt-in deployment policy.** The v0.1 draft sketched two topologies (dedicated mediator agent vs board-adapter cycle detector). **Decision**: the **board adapter is the default**, runs a lightweight cycle detector on the watcher's rearm path (composing with ┬د13.16.6 / ┬د13.16.7), and emits `MediationProposal` when a 2-cycle is detected. This is cheaper (no extra agent slot) and composes naturally with the existing server-emitted `Ack{delivered}` and board `decisions` panel. **However**, when the deployment includes a peer cycle (collab ظ¤ collab, where the ┬د13.9 tree has no asymmetric leader to override), a **third-party mediator agent** MAY be spawned for that specific cycle and the board adapter MAY defer to it; the spawn is a deployment-policy choice, not a wire-level requirement. **Reasoning**: the cost of server-coupling (board adapter knows about the ┬د13.19 application protocol) is acceptable because the board adapter already knows about ┬د13.13 (it emits `Ack{delivered}`) and ┬د13.16.7 (BOARD_STALE gate) ظ¤ the ┬د13.19 cycle detector extends a coupling that already exists, rather than introducing a new one. The third-party mediator opt-in covers the case where the peer-cycle's parties prefer not to involve the server in their dispute (e.g. for audit-isolation reasons in a future multi-tenant deployment).

**Q3 reference implementation ظ¤ OPT-IN, off by default.** The board-adapter cycle detector is shipped in the reference server (`runtime/server.cjs`) as an **opt-in** feature gated on `WS_DEADLOCK_DETECT` (default OFF); enable with `WS_DEADLOCK_DETECT=1` (+ optional `WS_DEADLOCK_PROBE_MS`, default 120000). When enabled, a periodic scan over the server's existing A2A reply-pairing map (`_a2aPending` ظ¤ the ┬د13.8 reply-window state) reconstructs lightweight wait-edges (`pending[B] = {from: A}` ظا║ `AظْB`, A awaits B's reply) and emits a `DeadlockProbe { cycleMembers, waitEdges, class: 'strict' }` to the boards (via `wsToBoards`) when a **strict 2-cycle** (Aظ¤B) persists past the probe threshold on both edges. The detector is **deliberately off by default** so adopters who don't want the serverظ¤application coupling pay nothing: the *canonical* detection layer remains the agent-side turn-start wait-edge DFS (┬د13.19.3) ظ¤ which sees each agent's true `WAITING` state, not the server's approximation. The opt-in server detector is a convenience for deployments that want a server-side safety net without each agent running the DFS; it detects only **strict** cycles (mutual pending past threshold), NOT **quasi-deadlocks** (behavioral waits with no cycle ظ¤ those have no server-observable structure and remain the ┬د13.19.4 SLA-discipline's domain). The reply-pairing approximation can false-positive on a long healthy wait, which the dual-edge threshold + the ┬د13.16.6 agent-active correction mitigate; deployments needing N-agent cycles or quasi-deadlock coverage stay on the agent-side DFS. EG ships the opt-in detector as a reference; production adopters tune the threshold (or spawn a third-party mediator agent per the hybrid) to their topology.

**Q4 ظ¤ Human-fallback threshold: bounded probe count ├ù wait-class severity, with alarm-fatigue cap at ~3-5 escalations/day per human.** The v0.1 draft used "timeout ├ù 2" informally. **Decision**: the rigorous formal threshold is `escalate_to_human(cycle) := (tier-3 mediation failed within mediation-rearm-interval) OR (NEEDS_HUMAN signal observed) OR (effective_elapsed > 2 ├ù class-timeout AND tier-1/2/3 all attempted)`. The *daily cap* on tier-4 emissions per human watcher is **3-5 escalations per 24h wall-clock** (configurable; default 4) ظ¤ beyond that, further `EscalationRequest` emissions are queued (not suppressed) and surfaced as a *meta-escalation* batch ("4+ unresolved escalations accumulating; human attention bottleneck"). **Reasoning**: the ┬د13.13 alarm-fatigue gating principle constrains how loud the escalation layer can be ظ¤ a human watching the live board has cognitive bandwidth that itself becomes a constraint, and the empirical ceiling of ~3-5 per day is the calibration boundary at which escalation signal-to-noise stays above 1. The cap is per-human, not per-cycle: a cycle that legitimately needs human attention will get it; the cap only prevents *cycle storms* from drowning out individually-important escalations. Composition: the cap interacts with the ┬د13.18 "recommend + proceed" discipline ظ¤ non-branching choices within an escalation are still resolved by the agent without consuming the human-cap budget.

**Q6 ظ¤ Renegotiate-SLA frequency limit: fixed cap N=2, then human-escalate.** The v0.1 draft offered three options (fixed cap, exponential backoff, human-escalate on Nth). **Decision**: a renegotiation may occur **at most 2 times** for a given inbound `msgId` (initial `ReviewSLAAck` is not a renegotiation; the first revised `eta` is renegotiation #1; the second revised `eta` is renegotiation #2). On the 3rd attempted renegotiation, the recipient emits `NEEDS_HUMAN { for: msgId, summary: 'SLA renegotiation cap reached' }` and the sender is authorized to invoke ┬د13.19.6 tier 4 directly. **Reasoning**: an exponential-backoff budget would itself become a quasi-deadlock vector (each renegotiation doubles the eta, so after a few rounds the eta exceeds any reasonable human review cycle and the deferral becomes effectively permanent). A fixed cap of 2 keeps the renegotiation surface honest (genuine schedule changes get accommodated twice; the third attempt is a stronger signal that the recipient lacks the context to estimate honestly, which IS a `NEEDS_HUMAN` condition). The cap unifies with consensus diff_adjusted_5 #3 (turn-cap + renegotiate-SLA unified) ظ¤ the turn-cap is the upper bound on the *count* of renegotiations, not on the *time* of each individual SLA. Composition: the cap interacts with ┬د13.19.5 (4) starvation guard ظ¤ repeated denial of `PreemptRequest` and repeated `ReviewSLAAck` renegotiation are tracked in *separate* counters; the starvation guard fires on the priority-denial counter, the renegotiation cap fires on the SLA-eta counter. Neither absorbs the other.

### 13.20 Blocker tracking + periodic nudge discipline ظ¤ making "waiting on external work" observable and time-bounded

Discipline (not a rule of nature): when ┬د13.16.6 element 5 (planned-queue scan) classifies *every* item as `blocked`, the agent is about to enter **true-exhaustion idle wait** ظ¤ the legitimate exception to "work continues" that ┬د13.16.6 reserves for the case where no progressable item exists. The failure mode ┬د13.20 prevents is what happens *after* that classification fires honestly: the agent waits on external parties (main / hub / a specific peer) who could resolve the blocker, but who do not progress on their side either ظ¤ and the wait silently extends across cycles without anyone (the agent, the human, the responsible party) seeing the cumulative cost. ┬د13.16.6 element 5 catches "I have planned work I am skipping"; ┬د13.20 catches "I am waiting on external work that is not progressing". They are paired turn-end controls at different layers ظ¤ element 5 is the *individual-agent* layer (am I idling while I have work?), ┬د13.20 is the *team-coordination* layer (am I idling while the team is also stalled on my behalf?). The ┬د13.20 layer is the silent-stall surface that ┬د13.16.6's individual-agent discipline cannot reach by construction (an agent that correctly has nothing to do is correctly idle; the team's failure to unblock it is not surfaced by the agent's queue scan alone).

#### 13.20.1 Status & provenance

- **Status**: discipline (P1 ظ¤ EG 1ه░ذهدهïإ contract). Pairs with ┬د13.16.6 (turn-end ritual) at the team-coordination layer and composes with ┬د13.19 (deadlock resolution) for the linear-but-stalled blocker class (distinct from ┬د13.19's cyclic-wait class ظ¤ see ┬د13.20.6).
- **Provenance**: user board UserPrompt `p-mptj6cxj`, 2026-05-31. The originating directive specified three discipline elements: (1) blocker-manifest emission before idle wait, (2) periodic nudge cadence during idle wait, (3) escalating intensity tied to nudge-count when the responsible party keeps deferring ("هئم╛╕ وـ┤هجءهـ╝ وـبم▒░ م╣îنذ╣ه£╝نر┤"). - **Failure mode this addresses**: agent enters "true-exhaustion idle wait" honestly (planned queue empty by classification, every item `blocked` per ┬د13.16.6 element 5), then sits across multiple rearm cycles while the external blocker remains unaddressed by the responsible party. From the agent's perspective the wait is legitimate (no progressable item exists in its own queue); from the *team's* perspective the wait is a silent stall (the responsible party either forgot, deprioritized, or never received a clear-enough nudge to act). #### 13.20.3 Pre-idle emit rule ظ¤ "make the wait observable before entering it"

When ┬د13.16.6 element 5 (planned-queue scan) classifies every queue item as `blocked` and the agent is therefore authorized to enter true-exhaustion idle wait, the agent MUST ظ¤ before emitting the idle-wait state and re-arming the watcher ظ¤ perform two emissions in this order:

1. **Emit the `BlockerManifest` `CUSTOM` frame to the board**, carrying every active blocker per ┬د13.20.2. The frame is a *snapshot*, not a delta ظ¤ it republishes the full set every time, so the board panel (┬د13.20.7) renders authoritatively from the latest snapshot without needing reconciliation logic. The frame is emitted via the agent's standard outbox path (not bridge-auto, per the ┬د13.13 no-auto-ack discipline carried forward).
2. **Append a 1-line summary in the agent's chat thread / response text**: `"Idle: N blockers (subject1 #count1, subject2 #count2, ...)"` ظ¤ minimal, single-line, surfaces the *fact* of the wait + the *shape* of the blocker set, without consuming user attention with manifest detail (the detail lives on the board panel for the user to drill into if they choose).

Both emissions are mandatory; neither is optional based on agent volition. Skipping the manifest emission while entering idle wait is the ┬د13.20 violation ظ¤ the wait then becomes the silent-stall failure mode this section exists to prevent. Skipping the chat-thread summary is the ┬د13.11 board-emission discipline failure recast at the ┬د13.20 surface ظ¤ the wait becomes invisible at the response-text channel where the human's attention naturally lands first.

**Composition with ┬د13.16.6 element 5**: element 5 classifies the queue and authorizes idle wait when classification returns "all blocked"; ┬د13.20.3 is the *additional* discipline that fires *only when* element 5's classification authorizes the wait. If element 5 returns "at least one progressable item" the agent takes that item and ┬د13.20.3 does not fire this turn (no idle wait to make observable). The two rules are sequenced, not alternative: element 5 first (do I have work I can do?), then ┬د13.20.3 if and only if element 5 cleared idle-wait entry (make the wait observable).

#### 13.20.4 Periodic nudge cadence ظ¤ `BlockerNudge` every N rearm cycles

During the idle wait that ┬د13.20.3 made observable, the agent does NOT just wait passively. **Default cadence**: `CONSTELLATION_BLOCKER_NUDGE_CYCLES = 1` ظ¤ emit one nudge per blocker per rearm cycle (i.e. roughly every ~45 min effective-elapsed, every cycle, the responsible party receives a nudge for every still-active blocker). #### 13.20.6 Composition with ┬د13.19 deadlock resolution

┬د13.20 and ┬د13.19 occupy adjacent but distinct surfaces. The decision rule:

- **┬د13.19 applies** when the wait pattern shows a **cyclic dependency** ظ¤ A waits on B, B waits on A (strict deadlock) or A waits on B who waits on C who waits on A (cycle of depth N). The detection signal is the wait-edge DFS returning a cycle (┬د13.19.3). Resolution flows through the ┬د13.19.5 primitives (preemption / leader override / mediation) and the ┬د13.19.6 ladder.
- **┬د13.20 applies** when the wait pattern is **linear-but-stalled** ظ¤ A waits on B, B is not waiting on A or on any cycle member; B simply hasn't progressed the work despite having the authority and inputs to do so. The wait-edge graph shows a single edge, no cycle. Resolution flows through ┬د13.20.4 nudges and the ┬د13.20.5 ladder.

**Shared infrastructure** ظ¤ both sections use the ┬د13.13 ack layer as the substrate:

- The ┬د13.13 server-emitted `Ack{delivered}` confirms wire-transport on both nudges and deadlock primitives.
- The ┬د13.19.7 `ReviewSLAAck` commitment ack is the *correct response* to a `BlockerNudge` tier-2 ETA query ظ¤ the responsible party honors the ┬د13.20 ladder by emitting a `ReviewSLAAck` with a concrete `eta`, converting the blocker from "stalled" into "scheduled". A `ReviewSLAAck` response to a `BlockerNudge` resets the ┬د13.20 tier to 1 (the responsible party has now committed; the cumulative-cost ladder pauses while the SLA window is honored) and re-engages the ┬د13.19.4 (B) SLA-OR-WORK discipline.
- The ┬د13.20 tier-4 escalation routes through the *same* ┬د13.19.6 tier-4 path (board `decisions` panel via ┬د13.17). ### 13.21 Fresh-context defer is an anti-pattern ظ¤ no "next session will pick it up" excuse

**The rule**: the agent MUST NOT defer planned work to "the next fresh context" / "after compact" / "next session will pick this up". When the planned-queue scan (┬د13.16.6 element 5) identifies a progressable item, the agent starts it **now, in the current context** ظ¤ not as a memo for an imagined future self that will arrive with clean token budget and pristine attention. The defer-to-fresh-context move is a *self-permissioned escape from the autonomous-execution surface*, and it MUST be treated as a discipline violation in the same tier as ┬د13.17 (forbidden main-chat structured-choice prompts) and ┬د13.18 (soliciting on non-branching choices). The agent does not get to opt out of work by predicting a more convenient time slot for the same work to be done.

**Why this is an anti-pattern ظ¤ the four-line argument**:

1. **Compact is not agent-controlled** ظ¤ the compact event fires when the harness decides the token budget threshold has been crossed; the agent has NO `/compact` discretion to schedule when context resets happen, and the agent has NO ability to *guarantee* the next inbound after compact will be the resumption of this exact work. The agent's expectation that "the next fresh context will pick this up" treats an event the agent cannot trigger as if the agent had triggered it.
2. **Fresh-context expectation is wishful + unrealistic** ظ¤ even if compact does fire, the post-compact agent inherits only the materialized artifacts (per ┬د13.16.6 element 6), the running todo state, and whatever the user re-supplies. The post-compact agent does NOT inherit the in-context understanding that *this specific deferred item was intended to be the next thing to do*. The defer is therefore a bet that (a) compact happens when convenient, (b) the post-compact agent re-derives the same priority from the same artifacts, and (c) no new inbound preempts the deferred item. All three are unreliable; the compound is wishful.
3. **Compact may fire mid-cycle and the agent must still RESUME** ظ¤ when compact fires during multi-cycle work, the correct discipline is to **resume the cycle from the ┬د13.16.6 element 6 materialized artifacts** ظ¤ the RRP / PM ledger / WORKLIST queue / spec anchor entries the pre-large-work materialization step put in place. The agent does NOT get to treat the post-compact wake as "the cycle ended, I'm free to pick something else". The cycle ends when the WORK ends, not when the *context* ends; context boundaries are an implementation detail of the harness, not a unit of work.
4. **Idle-wait for an imaginary fresh-context cycle is the silent-stall failure mode under a different name** ظ¤ ┬د13.16.6 element 5 already forbids idle-wait while progressable work exists. "Deferring to fresh context" is the same idle-wait dressed as a scheduling choice: the agent is sitting idle *now* on the theory that the work will be done *later* by an instance the agent cannot actually conjure. The ┬د13.20 blocker tracking discipline does not authorize this either ظ¤ the responsible party for the deferred work is the agent itself, so there is no external blocker; the only blocker is the agent's own choice to defer.

**The mechanism that makes resume possible ظ¤ ┬د13.16.6 element 6**: this section's rule (no defer-to-fresh-context) is enforceable only because the 6th turn-end ritual element (pre-large-work context materialization) puts the durable artifacts in place. The two sections are paired: element 6 is the *capability* (the work can be resumed cold because the artifacts exist), and ┬د13.21 is the *discipline* (the agent MUST resume, not defer, because defer is anti-pattern). One without the other is incoherent: capability without discipline allows the defer (artifacts exist but the agent chooses idle-wait); discipline without capability is impossible to comply with (agent is told to resume but has no artifact to resume from). Both ship together as policy seq 120.

**Policy chain ظ¤ the three sections that form one disciplined behavior**:

1. **┬د13.18 (recommend + proceed)** ظ¤ when the agent identifies a planned item, the recommendation IS the answer; the agent announces it in 1 line and proceeds. No solicitation of approval to proceed, no "should I do this next?" round-trip.
2. **┬د13.16.6 element 5 (planned-queue scan, policy #442)** ظ¤ at every turn-end, the agent mechanically scans the planned queue and starts the first progressable item *now*. Idle-wait is reserved for true exhaustion (every queue item classified `blocked`).
3. **┬د13.21 (anti-defer)** ظ¤ the agent does NOT route around the queue scan by deferring the progressable item to a future context. The defer-to-fresh-context move is forbidden; if context budget is tight, ┬د13.16.6 element 6 materializes the work to durable artifacts so the *same* cycle can resume post-compact ظ¤ the work does not change hands or get postponed; it crosses the boundary as live state, not as a memo.

The three sections form one continuous discipline surface: the recommendation makes the work concrete (┬د13.18), the queue scan starts it now (┬د13.16.6 #442), and the anti-defer rule (┬د13.21) closes the only remaining defection path ظ¤ "I'll do it later, in a cleaner context". Without ┬د13.21 the other two are toothless: the agent could acknowledge the recommendation, identify the planned item, and then announce "deferring to fresh context" ظ¤ which formally passes ┬د13.18 and ┬د13.16.6 element 5 (the agent did proceed, into idle-wait) while operatively defeating both.

**Failure mode this addresses ظ¤ observed pattern**: the agent, absorbed in synthesis or facing dense remaining work, emits a closing response that says some variant of "the next fresh-context cycle can pick up [X]" or "this is best handled after compact" or "queuing for the next session". That phrasing is the anti-pattern surface. The work named in [X] is, by definition, work the agent could have started in the *current* context (or, if too large for the current context, work the agent should have materialized via ┬د13.16.6 element 6 to survive the compact and resume in-flight). The deferral *as a scheduling choice* is the violation; the work itself is fine. The fix is mechanical: when the closing response is about to say "next fresh context will handle this", the agent re-routes to either (a) start the item now if context permits, or (b) materialize per element 6 and resume after compact ظ¤ never (c) defer-and-idle.

**Tone ظ¤ discipline + failure-mode framing, not a rule of nature**: ┬د13.21 exists because the agent's volitional self-permission to defer is the failure mode the rule prevents. The Constellation harness does not enforce this rule; it is an *agent-side discipline* that the agent itself must run on every closing-response synthesis. The ┬د13.16.6 element 6 materialization is the structural counterpart that makes the discipline survivable in practice; without it, the rule would be a demand the agent cannot meet (resume from what?). With it, the rule is a demand the agent CAN meet, and the only remaining question is whether the agent chooses to.

**Cross-links**:

- **┬د13.16.6 element 5** (planned-queue scan / policy #442) ظ¤ the upstream operational rule. ┬د13.21 is the explicit "why" anchor that the implicit no-defer assumption in element 5 has always depended on. Element 5 says "start a progressable item now"; ┬د13.21 spells out why "now" is not negotiable to "in the next context".
- **┬د13.16.6 element 6** (pre-large-work context materialization) ظ¤ the structural enabler. ┬د13.21's demand to resume rather than defer is only meetable because element 6 ensures the resume target exists as a durable artifact. The two are introduced together (policy seq 120) and ship together; deploying ┬د13.21 without element 6 would be a rule the agent cannot comply with.
- **┬د13.18** (recommend + proceed) ظ¤ the choice-surface counterpart. ┬د13.18 forbids soliciting approval on a non-branching choice; ┬د13.21 forbids deferring a planned item once the recommendation is announced. Same autonomous-execution principle, two surfaces.
- **┬د13.17** (main-chat structured-choice prompts FORBIDDEN) ظ¤ sibling discipline at the question-emission surface. ┬د13.17 forbids the inline option UI; ┬د13.21 forbids the inline "next context will handle it" defer. Both are agent-side disciplines closing escape paths from autonomous execution.
- **Seed Principle #16** (autonomous execution ظ¤ absolute) ظ¤ the seed-tier anchor. ┬د13.21 is the explicit operational form of Principle #16 for the *time-surface* (context-boundary defer), the same way ┬د13.18 is the operational form for the *choice-surface* (non-branching solicit) and ┬د13.16.6 element 5 is the operational form for the *queue-surface* (idle-while-progressable). All three surfaces share the same principle; ┬د13.21 closes the time-surface defection path.

**Provenance**: main upstream policy seq 120, msgId `m-mptjjxmo-119`, 2026-05-31. Originating context: the user surfaced the fresh-context defer pattern as the remaining escape from autonomous execution ظ¤ even after ┬د13.18 (recommend+proceed) and ┬د13.16.6 element 5 (planned-queue scan / #442) closed the choice-emission and queue-consumption surfaces, the agent could still defer planned work to "next context" and self-permission idle-wait via that phrasing. ┬د13.21 closes that surface; ┬د13.16.6 element 6 is the paired structural enabler that makes the resume-rather-than-defer rule meetable in practice.

#### 13.22.2 Two-level abstraction ظ¤ Level 1 (virtual PR, A2A-only) vs Level 2 (Github PR, provider-mediated)

The same wire vocabulary (┬د13.22.3) and state machine (┬د13.22.8) describe two physical realizations of a "PR." Which applies is a function of *who operates the source and target sides* and *what file-system access the receiving side holds*. `PRRequest.level` is an explicit `1 | 2` payload field; it defaults to `2` when absent.

- **Level 1 ظ¤ virtual PR (A2A-only path)**. No provider-side PR object is created. Applicable when source and target sides **share an operator** AND **both hold direct repo-level file-system access**. The receiving side (holding commit-push authority on the target repo) reads the `sourceContentRef` carried by `PRRequest`, applies the change directly to its local checkout of the target repo, commits, and pushes ظ¤ using A2A envelopes only for review / ack signaling. `gh pr create` is skipped; MERGED ظëة "commit pushed to the target's integration branch."
- **Level 2 ظ¤ Github PR (provider-mediated path)**. The receiving side invokes `gh pr create` against the target provider; the PR exists as a real provider object; review and merge flow through the provider UI / API, with A2A envelopes mirroring state. Applicable when source and target are operated by different parties, OR the PR is cross-org, OR the target's integration policy requires a provider-side PR object (required-status-checks, provider-tier audit trail).

**Selection rule**. When `category == trusted-mirror` AND operator-shares-target AND target-repo-direct-access both hold, **Level 1 is the default**; otherwise Level 2. The receiving side MUST validate the requested `level` against its capability matrix (direct push to `targetRepo`? operator-shared?) and MUST reject a mismatch with a capability-mismatch reason; the sender MAY retry with the other level. The ┬د13.22.8 state machine is `level`-discriminated: both levels traverse the same five logical states, but the transition triggers and the MERGED realization differ.

#### 13.22.4 Agent topology ظ¤ local-bridge command (adopted)

Three realizations of the PR-bot role: **(a) Dedicated A2A agent** (long-running `pr-bot` holding provider credentials; clean isolation but needs a credential store + deploy surface); **(b) Local-bridge command ظ¤ ADOPTED** (extend `local-bridge` with a `createPR` port running in the operator's IDE alongside their `gh`/`git` auth ظ¤ no separate credentials; the flow: agent emits `PRRequest` ظْ local-bridge invokes provider CLI [L2] or applies-commits-pushes locally [L1] ظْ emits `PRDraftReady`); **(c) Hybrid** (dedicated bot for cross-machine/separate-identity, local-bridge for same-operator; deferred). #### 13.22.5 Approval discipline ظ¤ trusted-mirror / standard (per-PR opt-in default) / emergency

The autonomous-execution external-publish gate is canonical: a PR to a repo other than the one the agent operates in is an external publish requiring user opt-in *unless* a pre-authorized policy specifies otherwise.

- **`trusted-mirror`** (pre-authorized) ظ¤ a spec-mirror PR where the diff is *already reviewed in the source repo* and the target is *the deterministic mirror surface*. The pre-authorization lives in `state.json.pr_system.trusted_mirrors[]` as `(sourceRepo, sourceContentRef) ظْ (targetRepo, targetContentRef)` pairings. A matching `PRRequest { category: 'trusted-mirror' }` is created without per-PR prompt; per-deployment policy MAY rate-limit (ظëج1 per pair per 24h) to catch runaway automation.
- **`standard`** (per-PR user opt-in ظ¤ **the default**) ظ¤ any PR not matching a trusted-mirror entry. The receiving side surfaces the request via the live board `decisions` panel (┬د13.17 ظ¤ the canonical structured-choice surface); the user explicitly approves; on approval the PR is created. This is the external-publish gate's default.
- **`emergency`** (out-of-band approval reference) ظ¤ for hotfix mirrors faster than the user can respond; carries `outOfBandApprovalRef` (a `decisions` msgId the user pre-approved), validated against the board's persistence. **`inline-section-mirror` invariants** (the canonical spec-mirror content form): (1) **`git add` isolation** ظ¤ after applying `body`, `git add <targetPath>` ONLY, never `-A`/`.` (a fallthrough commits every untracked file ظ¤ the dogfood defect); (2) **No branch-fetch**; (3) **Existence check** ظ¤ verify `targetPath` exists before append, else reject `inline-mirror-target-missing`; (4) **Dry-run `abs+exists` log** ظ¤ when gated OFF, log `abs=<resolved> exists=<bool>` so path defects surface pre-live. **`targetPath` base** = relative to the receiver's repo root (`git rev-parse --show-toplevel`), deterministic + portable; process-cwd / absolute / sub-tree bases are NOT used.

#### 13.22.6 Cross-repo content sourcing ظ¤ ╬▓ظ▓ mirror-branch-on-target (adopted) + ╬▒ small-diff fallback

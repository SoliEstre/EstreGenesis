- **v2.4.10** (2026-06-01) — **`Constellation.md §13.16.8` new sub-section: A2A communication priority discipline.** Constellation-only patch (frontmatter version v2.3.3 → v2.3.4). New §13.16.8 (~50 lines) inserted between §13.16.7 (BOARD_STALE gate) and §13.17 (main-chat ban), codifying a discipline applied whenever a state change (user steering / new task / external event such as a counterpart `Delegate`/`Report`/`Status`/`BlockerNudge` inbound) produces a work plan containing one or more outbound A2A messages. **The rule**: the agent (1) scans the new plan's action list to identify outbound-A2A items, (2) also identifies prerequisite work the counterpart depends on — commit-push of referenced docs (a `Delegate` payload that points to a doc URL is unusable until the doc is reachable at that URL), payload assembly, `state.json`/live-board updates the counterpart's decision depends on — and (3) moves both the message and its prerequisites to the head of the queue ahead of other planned actions; (4) other actions resume only after the message is emitted. **Two technical rationales**: (a) **drift-containment** — if EG updates a spec/state/doc before notifying the counterpart, the counterpart continues operating on a stale baseline for the entire unnotified-gap duration, risking superseded-baseline work + follow-up reconciliation cost; sending the message first bounds the drift window to round-trip latency (transport-bounded, typically orders of magnitude smaller than EG's queue depth) rather than EG's local-work duration. (b) **counterpart-start-time minimization** — when the new plan obligates the counterpart to start work (DB benchmarking, reference-runtime cp, RRP review, etc.), total wall-clock to cycle completion is `message-emit-time + counterpart-work-duration`; advancing emit-time directly shortens the cycle, and local EG actions that don't gate the message can run in parallel with the counterpart's work once the message is out but cannot recover the head-start lost by emitting late. **Symmetry**: applies to any role classified by §13.16 (`main` / `local` / `collab` / `upstream`) when that role initiates a cross-agent change — worker-to-main report, main-to-worker delegate, EG-to-upstream report, upstream-to-downstream steering all follow the same rule. **Composition with existing §13.x — no override of orthogonal gates**: §13.14 redaction still gates the message *content* (placeholder substitution, technical-only phrasing); §13.17 main-chat ban still gates the message *destination* (board-mediated route for structured Q/A); §13.20 `BlockerManifest` emission is itself an A2A communication and gets the same priority; destructive-operation authorization gates are not bypassed when the prerequisite work is itself destructive. **Composition with §13.16.6** turn-end / next-cycle ritual: cursor-tail probe (element 1) and next-cycle rearm (element 2, relocated to next-cycle start in v2.4.9) run on their own schedule and are independent of this discipline — the priority rule governs new outbound messages triggered by the current cycle's work plan, not the liveness-maintenance loop. Element 5 (planned-queue scan) interacts in the obvious way: when the scan classifies the next progressable item as an outbound A2A message (or as prerequisite work for one), the priority rule confirms it as head-of-queue. **Composition with §13.18 recommend+proceed**: this priority rule sits above §13.18 — even a 1-line "recommend+proceed" announcement is itself an outbound message in the main-chat surface; if its emission unblocks a counterpart (a `Status` update the counterpart's `ReviewSLAAck` was waiting on, a board-side `Decision` downstream agents will read), the 1-line emission takes priority over other planned actions per the same drift-containment + counterpart-start-time rationales. **Failure mode this prevents**: 5-action plan with action #3 = `Delegate` to worker executed in plan-order delays the worker's start by the wall-clock cost of actions 1-2 and risks stale-state work if action 2 touched a spec the worker reads; the priority rule reorders to prerequisite-of-3 → 3 → (1, 2, 4, 5 in natural order), giving the same final state with counterpart starting at the earliest possible moment and drift bounded by round-trip latency. **Scope discipline**: Constellation-only patch. Seed lib unchanged (seeds reference §13.16 by anchor, not by body). Protocol wire unchanged. Per `feedback_release_versioning_cadence.md` — in-flight patch tag cut, not catch-up.
- **v2.4.9** (2026-06-01) — **`Constellation.md §13.16.6` element-ordering refinement: watcher rearm (element 2) relocates from turn-end to next-cycle start.** Constellation-only patch (frontmatter version v2.3.2 → v2.3.3). The 6-element turn-end ritual composition is unchanged in scope; what changes is the execution point of element 2 (unconditional watcher rearm). Element 2 now executes as the **first action at the start of the next user-facing response** (cycle N+1's start), not at the close of the current response (cycle N's turn-end). Elements 1 (cursor-tail probe), 3 (board-state real-time update), 4 (BOARD_STALE gate handoff), 5 (planned-queue scan), and 6 (pre-large-work context materialization) continue to execute at turn-end. The mandatory-order constraint (element 1 strictly before element 2) still holds and now spans the cycle-N→cycle-N+1 turn boundary. **Technical rationale**: (a) **concurrent-watcher-overlap avoidance** — running element 2 at turn-end of cycle N while the prior cycle's watcher may still be polling produces two watchers armed at near-identical `base` cursors covering the same forward window; the same-turn rearm does not add coverage (both share the armed-window blind spot) but does add a redundant concurrent watcher. Relocating to next-cycle start spawns exactly one watcher at a time across turn boundaries — the prior watcher has either WAKE'd or REARM-timed-out by cycle N+1's start, so the next-cycle-start spawn replaces rather than overlaps. (b) **Cleaner base-cursor semantics** — element 1 at turn-end of cycle N advances the agent's cursor to include any inbound below the prior watcher's `base`; if element 2 also ran at turn-end of cycle N, its fresh watcher's `base` would still miss any inbound that arrives between probe-time and final-text-emission. Relocating element 2 to next-cycle start means the rearm `base` is the line count at cycle-N+1-start, which by construction already includes any inbound the cycle-N+1-start re-probe just surfaced; the rearm-base invariant becomes "cursor-after-latest-processed-inbound" rather than "cursor-at-an-arbitrary-mid-turn-moment". **Net effect**: same coverage with one watcher armed at a time across turn boundaries, cleaner base-cursor invariant. **What didn't change**: all other turn-end ritual elements stay at turn-end; the cost-asymmetry argument for unconditional spawn discipline still holds; element 1 strictly before element 2 still holds.
